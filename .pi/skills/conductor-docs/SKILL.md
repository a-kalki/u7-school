# Conductor Docs — Правила ведения документации

## Директива

Ты ДОЛЖЕН следовать этим правилам при **любых** изменениях документации в `conductor/`:
создании, редактировании, перемещении, удалении `.md`-файлов, а также при обновлении индексных файлов.

---

## 1. Главный принцип: легкость и полезность

**Документация не должна быть исчерпывающей.** Её цель — дать **высокоуровневые правила**
и **основу**, достаточную для начала работы. Детали агент может извлечь из живого кода.

- ✅ Правильно: 20–40 строк на файл, ключевые правила, 1–2 примера
- ❌ Неправильно: 200+ строк, описание каждого чиха, дублирование кода

---

## 2. Структура файла документации

Каждый файл документации для конкретного типа объекта (usecase, story, controller и т.д.) следует шаблону:

```markdown
# <Название> — Styleguide

**Назначение:** одно предложение — что это и где лежит в коде.

---

## 1. Ключевые правила

Список из 3–7 пунктов. Только то, что нельзя нарушать.

## 2. Структура / Пример

Минимальный пример кода (10–20 строк), показывающий костяк.

## 3. Связанные файлы

Ссылки на соседние styleguide-файлы.
```

---

## 3. Ссылки на живой код вместо дублирования

Если правило можно проиллюстрировать ссылкой на существующий файл в кодовой базе —
**используй ссылку**, а не копируй код в документацию.

- ✅ Правильно: `См. тесты: packages/user/src/api/user/register-guest-uc.test.ts`
- ❌ Неправильно: копировать 50 строк теста в .md

**Предпочтительные файлы для ссылок:**
- Тесты usecase (get и command-сценарии)
- Тесты story
- Реальные имплементации (не абстрактные классы)

---

## 4. Правила перемещения и удаления

### При удалении файла:
1. Найти **все ссылки** на него: `grep -rn "имя-файла" --include="*.md" conductor/ .pi/`
2. Обновить или удалить каждую ссылку
3. Убедиться, что `grep` больше не находит битых ссылок
4. Только потом удалить файл

### При перемещении файла:
1. Переместить файл
2. Обновить **все** ссылки во всех `.md`-файлах (включая `index.md`, `product-guidelines.md`, архивные spec.md)
3. Проверить битые ссылки

### При создании файла:
1. Добавить ссылку в соответствующий `index.md`
2. Если файл в `skills/` — добавить ссылку в карту соответствующего ddd-скилла

---

## 5. Объединение файлов

Если два файла описывают смежные темы — объедини их в один:

1. Выбери **один** файл как основной (обычно тот, что в `skills/` или более специфичный)
2. Перенеси уникальное содержимое из второго
3. Удали второй файл
4. Обнови все ссылки

---

## 6. Индексные файлы

`conductor/index.md` — главный индекс. Он должен содержать ссылки на все styleguide-файлы.
При добавлении/удалении файла — обнови `index.md`.

Индексы треков (`conductor/tracks/<id>/index.md`) должны указывать на spec.md, plan.md, summary.md.

---

## 7. Не раздувать

Если файл описывает каждый чих и превышает 80 строк — это сигнал к упрощению:

- Убрать очевидные вещи (агент может понять из кода)
- Заменить длинные примеры ссылками на живые тесты
- Оставить только правила, которые **реально можно нарушить**

---

## Предотвращение регресса

- **Не ломай существующие ссылки** — всегда проверяй `grep` после перемещения/удаления.
- **Не дублируй** — если информация уже есть в другом файле, лучше дай ссылку.
- **Индекс всегда актуален** — `index.md` должен отражать реальную структуру файлов.
