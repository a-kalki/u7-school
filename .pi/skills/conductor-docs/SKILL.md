# Conductor Docs — Правила ведения документации

## Директива

Ты ДОЛЖЕН следовать этим правилам при **любых** изменениях документации в `conductor/`:
создании, редактировании, перемещении, удалении `.md`-файлов, а также при обновлении индексных файлов.

---

## 1. Главный принцип: легкость и полезность

**Документация не должна быть исчерпывающей.** Её цель — дать **высокоуровневые правила** и **основу**, достаточную для начала работы. Детали агент может извлечь из живого кода.

- ✅ Правильно: 30–100 строк на файл, ключевые правила, возможны 1–2 примера, но лучше дать ссылки на файлы где можно посмотреть.
- ✅ Правильно: высокоуровневые принципы и правила, а не конкретные решения.
- ❌ Неправильно: 200+ строк, описание каждого чиха, дублирование кода, приведение кодового примера для каждого описываемого правила.

**Правила составления и ведения документации:**
  - описывать в основном: цели, архитектурные решения, ответственности, принципы, правила, соглашения.
  - не раздувать документацию, не дублировать одно и то же в разных файлах.
  - не делать документацию хрупким через чрезмерные детали, описывать высокоуровнево, для деталей дать явные ссылки в код;

---

## 2. Структура файла документации

Каждый файл документации для конкретного типа объекта (usecase, story, controller и т.д.) следует шаблону:

```markdown
# <Название> — Styleguide

**Назначение:** одно предложение — что это и где лежит в коде.

---

## 1. Ключевые правила

Высокоуровневые правила без кода. Список из 3–7 пунктов. Только то, что нельзя нарушать.

## 2. Соглашения по архитектуре

Принятые соглашения по архитектуре, best practics, можно со ссылками на файлы (код).

## 3. Тестирование

Как принято тестировать данный слой приложения, можно со ссылками на файлы (код).

## 3. Связанные файлы

Ссылки на соседние styleguide-файлы (при необходимости).
```

---

## 3. Ссылки на живой код вместо дублирования

Если правило можно проиллюстрировать ссылкой на существующий файл в кодовой базе — **используй ссылку**, а не копируй код в документацию.

- ✅ Правильно: `См. тесты: packages/user/src/api/user/register-guest-uc.test.ts`
- ❌ Неправильно: копировать 50 строк теста в .md

**Предпочтительные файлы для ссылок:**
- Реальные имплементации get, command сценариев (или абстрактные классы если это нужно по контексту)
- Тесты этих имплементации

---

## 4. Правила перемещения и удаления

### При удалении файла:
1. Найти **все ссылки** на него: `grep -rn "имя-файла" --include="*.md" conductor/ .pi/`
2. Обновить или удалить каждую ссылку
3. Убедиться, что `grep` больше не находит битых ссылок
4. Только потом удалить файл

### При перемещении файла:
1. Переместить файл
2. Обновить **все** ссылки во всех `.md`-файлах (включая `index.md`, `product-guidelines.md`, архивные spec.md)
3. Проверить битые ссылки

### При создании файла:
1. Добавить ссылку в соответствующий `index.md`
2. Если файл в `skills/` — добавить ссылку в карту соответствующего ddd-скилла

---

## 5. Индексные файлы

`conductor/index.md` — главный индекс. Он должен содержать ссылки на все styleguide-файлы.
При добавлении/удалении файла — обнови `index.md`.

Индексы треков (`conductor/tracks/<id>/index.md`) должны указывать на spec.md, plan.md, summary.md.

---

## 6. Не раздувать

Если файл описывает каждый чих и превышает 100 строк — это сигнал к упрощению:

- Убрать очевидные вещи (агент может понять из кода)
- Заменить длинные примеры ссылками на живые тесты
- Оставить только правила, которые **реально можно нарушить**

---

## 7. Предотвращение регресса

- **Не ломай существующие ссылки** — всегда проверяй `grep` после перемещения/удаления.
- **Не дублируй** — если информация уже есть в другом файле, лучше дай ссылку.
- **Индекс всегда актуален** — `index.md` должен отражать реальную структуру файлов.

## 8. Правила это не закон

Всегда руководствоваться здравым смыслом. Все правила можно нарушать если есть для этого причины. В подобных случаях всегда лучше остановиться и спросить пользователя.
