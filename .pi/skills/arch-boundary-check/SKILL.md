# Arch boundary check (u7-school, DDD-монорепо)

Цель: ловить момент, когда изменения проходят сквозь несколько слоёв — самое опасное место для потери архитектурного контекста.

## Когда запускаться
- **Да:** diff затрагивает файлы в `domain/` И `api/` одного пакета одновременно, изменения в `package.json` (поля `exports`/`imports`), новый файл facade или domain-service в любом пакете, изменения в `packages/core/src/` + изменения в любом другом пакете
- **Нет:** изменения строго внутри одного слоя, тесты, конфиги, форматирование

## Протокол разбора

### Шаг 1. Определи контекст
1. `git branch --show-current` → содержит `track_id`? → трек
2. Прочитай `conductor/tracks.md` → есть трек в `[~]`? → трек
3. `git log -1 --format="%H %s"` → есть коммит? → standalone с SHA
4. Иначе → standalone WIP с `git diff --stat`

Если изменения не подпадают под критерии «Да» — **пропусти дебриф**, не задавай вопросы.

### Шаг 2. Попроси нарисовать поток данных
Спроси: «Ты изменил файлы в разных слоях. Опиши цепочку: откуда приходит запрос (контроллер/bot/CLI) → какой UseCase → какой Aggregate → какой Repo → что возвращается обратно?»

### Шаг 3. Задай 2–3 вопроса по архитектурным границам
Выбери релевантные:

- **Слои:** логика, которая сейчас в UseCase — не должна ли она быть в Aggregate? И наоборот: то, что в Aggregate — не доменный ли это сервис?
- **Entry point:** новый код доступен через `index.ts` пакета? Правильно ли настроен `package.json` exports? Не создан ли плоский импорт в обход `#`?
- **Facade:** если другой модуль захочет использовать эту логику — какой путь? Нужен ли новый метод в фасаде?
- **DI:** новые зависимости зарегистрированы в `ModuleResolve`? Не создалась ли циклическая зависимость между пакетами?
- **Core changes:** изменения в `@u7/core` затронули все пакеты или только один? Не сломались ли `#`-импорты в других пакетах?
- **Domain service vs Aggregate:** новая логика — операция над одним агрегатом или над несколькими? Если над несколькими — она в domain-service?

### Шаг 4. Глубина
- `elii` → объясни разницу между слоями на примере нашего кода
- Уверенный ответ → «а как бы это выглядело, если бы мы использовали другой паттерн (например CQRS вместо Command)?»

### Шаг 5. Запиши результат
1. Создай лог-файл `data/debrief/logs/YYYY-MM-DD-<контекст>.md`:
```markdown
# Дебриф: YYYY-MM-DD

- **Контекст:** трек `<track_id>` / standalone
- **Skill:** arch-boundary-check
- **Коммит:** `<sha>` / WIP
- **Задача:** <описание>
- **Изменённые файлы:** <список>

## Вопросы и ответы

### 1. <вопрос>
**Мой ответ:** ...
**Оценка:** ✓ понял / ⚠ не уверен / ✗ не понял
```
2. Вызови скрипт:
```bash
./.pi/skills/post-task-debrief/add-debrief.sh "<дата>" "<трек или standalone>" "<sha или WIP>" "arch-boundary-check" "<итог>" "logs/<имя-лог-файла>"
```
3. Вызови `./scripts/maintenance-debrief.sh`, выведи результат.

Я назвал цепочку прохождения запроса, объяснил почему логика в конкретном слое, и определил точку входа для других модулей.
