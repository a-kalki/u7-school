# Content integrity check (u7-school, учебный контент)

Цель: не дать сломать структуру курсов при генерации или правке контента. Платформа u7-school — это в первую очередь контент, и его целостность критична.

## Когда запускаться
- **Да:** изменения в `data/lessons/`, `data/courses/`, `packages/w3school/output/`, файлы `steps.json`, `modules.json`, `lessons.json`, `lesson.md`, `summary.md`, mentor-файлы
- **Нет:** изменения в `*.ts`, конфигах, тестах

## Протокол разбора

### Шаг 1. Определи контекст
1. `git branch --show-current` → содержит `track_id`? → трек
2. Прочитай `conductor/tracks.md` → есть трек в `[~]`? → трек
3. `git log -1 --format="%H %s"` → есть коммит? → standalone с SHA
4. Иначе → standalone WIP с `git diff --stat`

Если изменения не подпадают под критерии «Да» — **пропусти дебриф**, не задавай вопросы.

### Шаг 2. Попроси перечислить затронутый контент
Спроси: «Какие конкретно уроки, шаги или модули ты изменил? В каком проекте и модуле они находятся?»

### Шаг 3. Задай 2–3 вопроса по правилам контента
Выбери релевантные:

- **Нумерация:** папки уроков соответствуют шаблону `p<N>-l<M>-<kebab-name>`? Правильно ли обновлены `lessonIds` в проекте внутри `modules.json`?
- **Шаблонные переменные:** нет ли `$T1`, `$T2`, `$T3` в `steps.json`, `lesson.md`, `summary.md`? (правило из AGENTS.md — запрещены)
- **Mentor-файлы:** 20–30 строк? 10–15 `console.log`? Нет вопросов-подсказок в коде? Если больше — разбит ли на несколько файлов? (правило из `conductor/guides/lesson-design.md`)
- **Видео-ссылки:** оформлены как `[p<N>-l<M>. Название.mp4](URL)`? URL полный, без переменных?
- **UUID:** все новые уроки имеют UUID? UUID добавлен и в `lessons.json`, и в `lessonIds` проекта?
- **Шаги:** каждый шаг имеет `type` (student/mentor), `content` без переменных, правильный `stepNumber`?
- **Кросс-ссылки:** нет ли битых ссылок между уроками и шагами? Все `stepId` в уроке ссылаются на существующие шаги?

### Шаг 4. Глубина
- `elii` → объясни структуру контента на примере одного урока
- Уверенный ответ → «а что будет, если ментор захочет вставить этот урок в другой модуль — что сломается?»

### Шаг 5. Запиши результат
1. Создай лог-файл `data/debrief/logs/YYYY-MM-DD-<контекст>.md`:
```markdown
# Дебриф: YYYY-MM-DD

- **Контекст:** трек `<track_id>` / standalone
- **Skill:** content-integrity-check
- **Коммит:** `<sha>` / WIP
- **Задача:** <описание>
- **Изменённые файлы:** <список>

## Вопросы и ответы

### 1. <вопрос>
**Мой ответ:** ...
**Оценка:** ✓ понял / ⚠ не уверен / ✗ не понял
```
2. Вызови скрипт:
```bash
./.pi/skills/post-task-debrief/add-debrief.sh "<дата>" "<трек или standalone>" "<sha или WIP>" "content-integrity-check" "<итог>" "logs/<имя-лог-файла>"
```
3. Вызови `./scripts/maintenance-debrief.sh`, выведи результат.

## Готово, когда
Я назвал все затронутые файлы контента, подтвердил соответствие правилам нумерации, отсутствие шаблонных переменных и корректность mentor-файлов.
