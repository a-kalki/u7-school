# Post-task debrief (u7-school, DDD/TypeScript/Bun)

Цель: держать понимание кодовой базы в тонусе. После завершения задачи становлюсь кратким ревьюером-преподом на ~5 минут. Инкрементально, не стеной в конце.

## Когда запускаться

### Да
Есть особо важные объекты за изменениями которых нужно особо пристально следить:
- сущность (entity);
- политика (policy);
- агрегат (aggregate);
- репозиторий (repo);
- доменный сервис (domain-service);
- сценарий использования (use-case);
- межмодульные взаимодействия (facade);
- пользовательские сценарии (user-story);
- контроллеры (controller);
- запуск приложения (app, main, bot);

Нужно уметь находить баланс в изменениях:
- насколько масштабны изменения;
- насколько влияют на поток выполнения;
- какова цена ошибок;

### Нет
- переименования;
- форматирование;
- комментарии;
- правки только в тестовых файлах;
- косметические или незначительные изменения в коде;
- изменения без кода, правки в `*.md`, `*.json` конфигах.

## Протокол разбора

### Шаг 1. Определи контекст
1. `git branch --show-current` → содержит `track_id`? → трек
2. Прочитай `conductor/tracks.md` → есть трек в `[~]`? → трек
3. `git log -1 --format="%H %s"` → есть коммит? → standalone с SHA
4. Иначе → standalone WIP с `git diff --stat`

Если изменения не подпадают под критерии «Да» — **пропусти дебриф**, не задавай вопросы. Заверши выполнение скилла без записи.

### Шаг 2. Попроси пересказать
Спроси меня своими словами: что изменилось и зачем. Я отвечаю первым — ты НЕ раскрываешь ответ. Не принимай «да, я понял» без конкретики.

### Шаг 3. Возьми реальный кусок диффа
Выбери ОДИН ханк из `git diff` и спроси: «почему здесь так, а не иначе?»

### Шаг 4. Задавай вопросы по граничным случаям НАШЕГО стека
Не задавай больше 3 вопросов за раз. Не двигайся дальше, пока я не закрою текущий пункт.

### Шаг 5. Глубина
- Команда `elii` → объясняй как интерну, на пальцах
- Уверенный ответ → копай на уровень глубже: «а почему не альтернативный подход X?»

### Шаг 6. Запиши результат
1. Создай лог-файл `data/debrief/logs/YYYY-MM-DD-<контекст>.md`:
```markdown
# Дебриф: YYYY-MM-DD

- **Контекст:** трек `<track_id>` / standalone
- **Skill:** post-task-debrief
- **Коммит:** `<sha>` / WIP
- **Задача:** <описание>
- **Изменённые файлы:** <список>

## Вопросы и ответы

### 1. <вопрос>
**Мой ответ:** ...
**Оценка:** ✓ понял / ⚠ не уверен / ✗ не понял
```
2. Вызови скрипт для добавления записи в реестр:
```bash
./.pi/skills/post-task-debrief/add-debrief.sh "<дата>" "<трек или standalone>" "<sha или WIP>" "post-task-debrief" "<итог например 3/3 ✓>" "logs/<имя-лог-файла>"
```
Скрипт сам добавит строку в `data/debrief-registry.md` и обновит счётчик.

3. Вызови скрипт обслуживания — проверит, не пора ли graduation или ревизия:
```bash
./.pi/skills/post-task-debrief/maintenance-debrief.sh
```
Выведи результат пользователю. Если скрипт предлагает graduation — обсуди.

## Готово, когда
Я объяснил проблему, решение, ключевое дизайн-решение и хотя бы один edge case. Итоговая оценка записана в лог и реестр.
