# Руководство по продукту (Guidelines)

## UX и Навигация
- Приложение в перспективе предоставляет ui/ux по трем каналам (по приоритету реализации):
    - консоль: для админов и частично менторов, позволит быстро создавать контент;
    - бот: основной канал работы студентов и менторов при занятиях;
    - web-ui: дополнительный канал работы при необходимости;
- Принципы ui/ux:
    - структурность и простота: пользователь по иерархии ui должен легко находить нужный функционал;
    - пошаговость: лучше избегать создания за раз сложных объектов, лучше создавать по шагам с промежуточным сохранением данных;
    - понятность: придерживаться минимализма и дозированности, лучше избегать выдачу большого объема информации за раз. Особенно это касается текстовых ui (консоль, бот).

## Обработка ошибок
- **Явное информирование:** Бот не должен скрывать ошибки, он должен понятно сообщать о том, что произошло.
- **Помощь ментора:** При необходимости бот должен направить студента к ментору.

## Оформление текста
- **MarkdownV2 Акценты:** Использование возможностей форматирования Telegram (жирный шрифт, курсив, моноширинный текст для кода) для акцентирования внимания на важных учебных терминах.
- **Эмодзи-навигация:** Применение стандартизированных эмодзи в меню и кнопках для быстрой визуальной ориентации (например, 📚 для курсов, 👤 для профиля).

## Нумерация контента курса

### Именование папок уроков

Все папки уроков в `data/fullstack-js/<модуль>/` именуются по шаблону:

```
p<номер_проекта>-l<номер_урока_в_проекте>-<короткое-название>
```

- `p` — номер проекта (1, 2, 3, 4...)
- `l` — номер урока внутри проекта (1, 2, 3...)
- Название — kebab-case, отражающее тему урока

**Примеры:**
- `p1-l1-introduction-to-programming` (проект 1, урок 1)
- `p2-l3-data-structures-array-set` (проект 2, урок 3)
- `p4-l5-recursion` (проект 4, урок 5)

### Видео-ссылки в контенте

В текстах уроков (`lesson.md`) и шагов (`steps.json`) видео-ссылки оформляются:

```
[p<проект>-l<урок>. Название урока.mp4](https://drive.google.com/...)
```

**Пример:** `[p1-l1. Введение в программирование.mp4](https://drive.google.com/file/d/xxx)`

### Порядок уроков в проекте

Порядок уроков внутри проекта определяется порядком `lessonIds` в массиве `projects` в `data/courses/modules.json`.

### Создание нового урока

1. Определи проект, к которому относится урок (см. `modules.json`)
2. Определи номер урока внутри проекта (следующий после последнего в `lessonIds`)
3. Создай папку по шаблону `p<N>-l<M>-<kebab-name>`
4. Добавь UUID урока в `lessons.json` и ссылку в `lessonIds` нужного проекта
5. Видео-ссылку оформи с новым номером

## Архитектурные правила разработки
Для реализации использовать следующие узкоспециализированные гайдлайны:

- [Общая архитектура](./code_styleguides/architecture.md) — **Выжимка:** слои модуля (domain/api/ui), entry point Module, поток выполнения команды (Console/Bot/REST -> Module -> UseCase -> Domain -> Repo), **Layer-based импорты** (exports/imports в package.json, `#`-импорты внутри пакета, запрет плоских импортов). **Использовать:** при проектировании нового модуля, выборе слоя для новой логики, понимании жизненного цикла команды, настройке импортов/экспортов пакета.

- [Backend-слой (UseCase, Module)](./code_styleguides/api.md) — **Выжимка:** ответственность UseCase (валидация, оркестрация) и Module (диспетчеризация, DI), их взаимосвязь, поток обработки запроса от контроллера до результата. **Использовать:** при реализации нового сценария использования (use-case), настройке модуля, добавлении точки входа.

- [Принципы DDD (Aggregate, Schema, Policy)](./code_styleguides/ddd.md) — **Выжимка:** Entity, Schema, Aggregate (инварианты), Policy (права доступа), ModuleMeta (метаданные модуля), правило наследования валидации в DTO. **Использовать:** при создании/изменении доменной модели, агрегатов, политик и схем валидации.

- [Обработка ошибок и исключений](./code_styleguides/skills/errors.md) — **Роль:** иерархия AppError (kind→HTTP), типизированные хелперы, обработка на контроллере через fromError(). **Использовать:** при выбросе бизнес-ошибок и форматировании ответа с ошибкой.
- [База известных проблем (skill `troubleshoot`)](../.pi/skills/troubleshoot/SKILL.md) — **Выжимка:** типичные ошибки библиотек и инструментов (valibot, bun, typescript, biome) и способы их решения. **Использовать:** при столкновении с неожиданной ошибкой — загрузи skill `troubleshoot`.

- [Инструкция по наполнению уроков шагами](./guides/lesson-design.md) — **Выжимка:** правила создания студенческих и менторских шагов, структура уроков, процесс утверждения. **Использовать:** при генерации шагов, наполнении уроков контентом, создании ментор-файлов.

- [Соглашения по именованию](./code_styleguides/naming.md) — **Выжимка:** kebab-case для файлов, шаблоны имён классов (Ar, Uc, Schema, Command), структура ModuleCommand. **Использовать:** при создании новых файлов, классов, папок и определении структуры команды модуля.

- [Правила тестирования](./code_styleguides/testing.md) — **Выжимка:** TDD (Red-Green), пирамида тестов (домен -> repo -> uc -> module -> e2e), изоляция и размер тестов, проверка AppException по name/kind, защита от регресса (запрет правок чужого кода без разрешения). **Использовать:** при написании любых тестов — от модульных до интеграционных.

- [Межмодульное взаимодействие и DI](./code_styleguides/integration.md) — **Выжимка:** инверсия зависимостей через ModuleResolve, правила импорта между модулями (через entry point, а не прямой вызов use-case), фасады для частого доступа. **Использовать:** при организации зависимостей модуля, доступе к данным соседнего модуля, настройке DI-контейнера.

- [TypeScript Styleguide](./code_styleguides/typescript.md) — **Выжимка:** настройки TypeScript, стиль кода, форматирование через Biome, правила типизации. **Использовать:** при написании кода на TypeScript для соблюдения единого стиля и настроек компилятора.
