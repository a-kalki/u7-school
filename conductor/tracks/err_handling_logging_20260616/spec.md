# Спецификация: Унификация обработки ошибок и логирования в bot-слое

## Обзор
Рефакторинг обработки ошибок и логирования на уровнях botRouter → bot-controller → bot-story.
Замена ad-hoc `console.error` на структурированное логирование через `this.appApi.logger`,
проверка, что контроллер перехватывает необработанные ошибки story,
аудит всех story-классов, документирование подхода в conductor.

## Функциональные требования

### FR1: Логгер в U7BotUserStory.handleError
- Добавить доступ к `this.appApi.logger` в `U7BotUserStory`.
- Заменить `console.error` в `handleError()` на `this.appApi.logger.error('bot', ...)`.
- Логировать ошибки с kind `internal`, `unauthorized` и **`default`** (неизвестный тип).
- Ошибки `validation`, `not-found`, `conflict`, `access-denied`, `bad-request` — **не** логировать
  (они часть нормального потока).

### FR2: Логгер в BotUserStory (core)
- Добавить поле/метод `logger` в базовый класс `BotUserStory` (пакет `core`),
  чтобы все наследники имели доступ к логгеру без дублирования.
- Логгер должен браться из `this.appApi.logger`.

### FR3: Проверка перехвата ошибок на уровне контроллера
- Проверить, что bot-controller перехватывает необработанные ошибки из story
  (те, что не обёрнуты в try/catch на уровне story).
- Если контроллер уже это делает — в story не нужно оборачивать каждый вызов `moduleApi.execute()`.
  try/catch в story остаётся только там, где нужна специфичная реакция на ошибку
  (например, показать пользователю сообщение об ошибке валидации).
- При необходимости — добавить спец-метод обработки ошибок в `core:BotController`.

### FR4: Аудит базовых классов core (только документирование)
- Проверить `BotUserStory` — все ли места обработки ошибок используют унифицированный подход.
- Проверить `ApiModule.dispatch()` — логирует ли ошибки, есть ли ad-hoc try/catch.
- Проверить `BaseUc` / `UseCase` — какие ошибки пробрасываются, есть ли подавления.
- **Без изменения кода.** Результат зафиксировать в аудит-заметке трека.

### FR5: Аудит всех story-файлов
Проверить каждый `*.story.ts` в проекте по критериям:
1. `this.handleError(err)` используется вместо ad-hoc логики.
2. Нет подавленных ошибок (пустой catch без обработки).
3. try/catch — только где нужна реакция на ошибку (не везде).

Область аудита:
- `packages/stream/src/ui/bot/stories/*.story.ts`
- `packages/course/src/ui/bot/stories/*.story.ts`
- `packages/user/src/ui/bot/stories/*.story.ts`
- `packages/onboarding/src/ui/bot/stories/*.story.ts`
- `packages/app/src/ui/stories/*.story.ts`

Для каждой story, где обнаружено нарушение — исправить обработку ошибок.

### FR6: Тесты на handleError (по здравому смыслу)
- Написать разумный минимум тестов, проверяющих, что:
  - `validation` ошибка показывает поля пользователю.
  - `internal` / `default` ошибка не раскрывает деталей.
- Тесты — только для代表性的 story (1–2), а не для каждой.

### FR7: Документация в conductor
- Обновить/создать документацию в `conductor/code_styleguides/`:
  - Обработка ошибок в story (когда нужен try/catch, когда нет).
  - Обработка ошибок в controller (перехват на верхнем уровне).
  - E2E-тестирование ошибок (возможно).
- Задокументировать принятый подход: ошибки, не пойманные в story, перехватываются контроллером.

## Нефункциональные требования
- NFR1: Тесты должны проходить в CI-режиме (`CI=true bun test`).
- NFR2: Покрытие нового/изменённого кода >80%.
- NFR3: Код должен проходить `bun run lint` и `bun run tslint`.

## Критерии приёмки
- [ ] `console.error` в `handleError()` заменён на `this.appApi.logger.error(...)`.
- [ ] `internal`, `unauthorized`, `default` ошибки логируются; остальные — нет.
- [ ] `BotUserStory` в core предоставляет поле `logger` из `appApi`.
- [ ] Подтверждено, что контроллер перехватывает необработанные ошибки story.
- [ ] Аудит core-классов выполнен, результаты задокументированы.
- [ ] Все `*.story.ts` проверены; где нужно — исправлены (убраны лишние try/catch, заменены ad-hoc обработчики).
- [ ] Есть 1–2代表性的 теста на `handleError`.
- [ ] Документация в conductor обновлена/создана.
- [ ] `bun test --coverage` показывает >80% покрытия изменённого кода.

## За рамками
- Изменения в `ApiModule`, `BaseUc`, `UseCase` (только аудит).
- Логирование на уровне инфраструктуры (БД, HTTP-клиенты).
- Обработка ошибок в CLI-приложении.
