# Спецификация: Улучшение CLI — авторизация и UX

## Обзор
Улучшить пользовательский опыт консольного приложения `apps/u7-cli`:
добавить механизм авторизации (бутстрап + логин), параметризовать
команду `list-users`, улучшить подсказки при вводе команд
и формат сообщений об ошибках валидации.

## Функциональные требования

### FR1: Бутстрап-регистрация первого администратора
- Команда `/register` доступна **только** когда репозиторий пользователей пуст.
- При запуске CLI, если пользователей нет — приложение предлагает `/register`.
- `/register` запрашивает необходимые поля и создаёт первого пользователя с ролью `Admin`.
- После успешной регистрации пользователь автоматически становится активным актором.

### FR2: Логин (выбор активного пользователя)
- Команда `/login` доступна всегда (кроме случая пустого репо).
- При запуске CLI, если пользователи есть — приложение предлагает `/login`.
- `/login` показывает список пользователей для выбора (или поиск по имени/id).
- Выбранный пользователь становится активным актором — его `actorId` подставляется во все последующие команды.
- Активный пользователь отображается в приглашении CLI.

### FR3: Параметризация `list-users`
- Команда `list-users` модуля `user` должна поддерживать параметры:
  - **Лимит** (`limit`): количество записей (по умолчанию 20).
  - **Фильтр по роли** (`role`): Admin, Mentor, Student и т.д.
  - **Поиск по имени** (`name`): частичное совпадение.
  - **Фильтр по telegramId** (`telegramId`): точное совпадение.
  - **Сортировка** (`sort`): поле и направление (например, `createdAt:desc`).

### FR4: Упреждающие подсказки при вводе команд
- При вводе пути к usecase'у (например, `/user/list-users`) приложение показывает
  приглашение с форматом ввода, где для каждого параметра указано:
  - Текущий формат (блок кода с путём и названиями полей) — сохранить.
  - Вне блока кода: название параметра, ожидаемый тип, допустимые значения перечислений,
    формат ввода массива.
  - Пример правильного ввода (если можно сгенерировать из схемы).

### FR5: Улучшенные сообщения об ошибках валидации
- При ошибке валидации выводить структурированную информацию из объекта ошибки Valibot:
  - Какое поле содержит ошибку.
  - Что ожидалось (тип, формат, допустимые значения).
  - Фактическое переданное значение (если применимо).
- Функция `formatValibotErrors()` размещается в слое `auto-ui/controller/` пакета `@u7/core`.
- Сообщение форматируется в `AutoUiConsoleController`, а не в слое `api`.

## Критерии приёмки
- [ ] При первом запуске CLI с пустой БД приложение предлагает `/register`.
- [ ] `/register` создаёт администратора и делает его активным актором.
- [ ] При повторном запуске CLI приложение предлагает `/login`.
- [ ] `/login` позволяет выбрать пользователя, его `actorId` подставляется в команды.
- [ ] `list-users` поддерживает все заявленные параметры фильтрации и сортировки.
- [ ] При вводе `/user/list-users` показывается подсказка со всеми параметрами и примером.
- [ ] Ошибка валидации показывает конкретное поле и причину в читаемом виде.
- [ ] Все существующие тесты проходят.

## За рамками
- Полноценная система ролей и прав доступа (RBAC).
- Аутентификация через внешние провайдеры (Telegram, OAuth).
- Персистентность сессии (переживание перезапуска CLI без повторного логина).
- Пагинация с cursor-based подходом.
