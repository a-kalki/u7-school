# Спецификация: Реализация функционала пользователя и курсов (domain + api + console)

## Обзор

Трек расширяет существующую доменную модель платформы, добавляя агрегаты (Ar), политики (Policy),
сценарии использования (Uc) и консольный интерфейс для управления пользователями и курсами.
Одновременно фиксируются архитектурные соглашения в проектной документации.

**Исходная база:** В пакете `@u7/core` уже определены:
- Тип `User` и схема `UserSchema` (поля: uuid, name, telegramId, role, createdAt, updatedAt?)
- Тип `Course` и схема `CourseSchema` (поля: uuid, title, description, authorId, status, ...)

## Функциональные требования

### FR-0: Соглашения об именовании
- **Сущность (тип):** `User`, `Course`
- **Схема валидации:** `UserSchema`, `CourseSchema`
- **Агрегат:** `UserAr`, `CourseAr` — принимает состояние сущности, определяет логику обновления
- **Политика прав:** `UserPolicy`, `CoursePolicy`
- **Объект БД (опционально):** `UserRecord`, `CourseRecord`
- **Объект вывода (опционально):** `UserOutput` / `UserOut`
- **Команда:** `CreateUserCommand`, `CreateCourseCommand`
- **Сценарий использования:** `UserCreatingUc`, `CourseCreatingUc`

### FR-1: Документация архитектуры
- **FR-1.1:** Документ `docs/architecture.md`, описывающий:
  - Двухслойную архитектуру бэкенда (domain + api).
  - Соглашения об именовании (FR-0).
  - Устройство модулей, эндпойнтов, команд и сценариев использования.
  - Правила межмодульного взаимодействия (фасады, прямые импорты политик).
  - Подход к инверсии зависимостей (DI), рассмотреть предлагают ли что либо elysia.
- **FR-1.2:** Зафиксированные решения:
  - Богатая доменная модель: агрегаты (Ar) инкапсулируют поведение.
  - Policy-объекты для прав доступа.
  - Инварианты встроены в схемы валидации Valibot.
  - Use-case ориентированный слой приложения.

### FR-2: Агрегат пользователя (UserAr)
- **FR-2.1:** `UserAr` принимает `User` как состояние.
- **FR-2.2:** Инкапсулирует логику изменения состояния (методы обновления полей с проверкой
  инвариантов через UserSchema).
- **FR-2.3:** Фабричный метод `UserAr.create(command)` — создаёт новый агрегат из команды.

### FR-3: Агрегат курса (CourseAr)
- **FR-3.1:** `CourseAr` принимает `Course` как состояние.
- **FR-3.2:** Инкапсулирует логику изменения (смена названия, описания, назначение автора).
- **FR-3.3:** Фабричный метод `CourseAr.create(command)`.

### FR-4: Политики (Policy)
- **FR-4.1:** `UserPolicy` — проверяет права на создание/чтение/редактирование пользователей.
- **FR-4.2:** `CoursePolicy` — проверяет права на создание/чтение/редактирование курсов.
- **FR-4.3:** Политики импортируются напрямую там, где нужна проверка прав.

### FR-5: Слой приложения — Use Cases
- **FR-5.1:** `UserCreatingUc`:
  - Принимает `CreateUserCommand` и `UserPolicy`.
  - Проверяет права через политику (кроме bootstrap-режима, см. FR-6.3).
  - Создаёт `UserAr` через фабричный метод.
  - Сохраняет через `UserRepository`.
  - Возвращает результат.
- **FR-5.2:** `CourseCreatingUc`:
  - Принимает `CreateCourseCommand`, `CoursePolicy`, `UserRepository`.
  - Проверяет права и существование автора.
  - Создаёт `CourseAr`.
  - Сохраняет через `CourseRepository`.
  - Возвращает результат.
- **FR-5.3:** Интерфейсы репозиториев (`UserRepository`, `CourseRepository`) с in-memory
  реализацией на данном этапе.

### FR-6: Эндпойнт модуля и консольный интерфейс
- **FR-6.1:** Единый эндпойнт `api/core` принимает команды и маршрутизирует их к нужному UC.
- **FR-6.2:** Команды — это простые объекты (plain objects), сериализуемые для передачи через
  разные UI (консоль, бот, REST).
- **FR-6.3:** Консольная команда `create-user`:
  - Принимает параметры (name, telegramId, role).
  - **Без проверки прав** (bootstrap-режим: в начальный момент в системе нет объектов).
  - Выводит результат.
- **FR-6.4:** Консольная команда `create-course`:
  - Принимает параметры (title, description, authorId).
  - **С проверкой прав** (к этому моменту администратор уже создан).
  - Выводит результат.

### FR-8: Архитектура межмодульного взаимодействия
- **FR-8.1:** Приложение делится на доменные модули (core, w3school, ...).
- **FR-8.2:** Межмодульные вызовы на чтение, изменение состояния — только через фасад модуля.
- **FR-8.3:** Политики можно импортировать напрямую для проверки прав.
- **FR-8.4:** Предусмотреть механизм DI для получения зависимостей (репозиториев, политик) без
  прокидывания через всю цепочку вызовов.

## Открытые вопросы

- **DI-механизм:** Как именно реализовать инверсию зависимостей? Варианты:
  - A. Простой контейнер (Map) в корне модуля, наполняемый при старте.
  - B. Функциональная инъекция (замыкания/каррирование).
  - C. Легковесный DI-контейнер.
- **Фасады vs прямые импорты:** Уточнить, всегда ли изменение состояния идёт через фасад, или
  есть исключения (например, внутри одного модуля).
