# Спецификация трека Stateless Bot & Onboarding Refactoring

## 1. Обзор
Цель трека — рефакторинг домена `Onboarding` для упрощения логики UI и реализация полностью "тупого" (stateless) Telegram-бота. Логика управления состоянием анкеты (черновики, переходы) переносится в агрегат. Также усиливается безопасность модуля `User` и автоматизируется выдача ролей.

## 2. Функциональные требования
### Домен и API
- **Рефакторинг агрегата**: `QuestionnaireAr` управляет всеми переходами и черновиками через единый метод `handleAction`.
- **Автоматизация ролей**: При завершении анкеты UseCase автоматически вызывает `UserFacade` для добавления роли `STUDENT`.
- **Безопасность User**:
    - `EnsureUserWithRoleUc` требует авторизации (ADMIN) и не делает авторегистрацию.
    - `RegisterGuestUc` — новый UC для явной регистрации GUEST ботом при `/start`.

### UI (Бот и Контроллер)
- **Игнорирование групповых чатов**: Бот работает только в приватных чатах.
- **Удаление локального состояния**: Бот не использует `conversations`.
- **Типизированный контракт**: `BotUpdate` (вход) и `BotResponse` (выход: `sendMessage`, `editMessage`).
- **Чистый Контроллер**: `OnboardingController` только транслирует ввод в `HandleOnboardingActionUc` и преобразует его результат в инструкции для бота.
- **Поддержка /cancel**: Сброс состояния в домене через API.

## 3. Критерии приёмки
- Тесты агрегата покрывают все сценарии `handleAction` (текст, одиночный выбор, множественный выбор с "Далее").
- `HandleOnboardingActionUc` успешно выдает роль студента по завершении.
- Бот корректно проходит сценарий анкеты, обновляя сообщения при выборе (edit) и отправляя новые при переходах (send).
- Попытки использовать `ensureUserWithRole` без прав ADMIN отклоняются.
