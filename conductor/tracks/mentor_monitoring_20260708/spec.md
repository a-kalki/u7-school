# Спецификация трека: Мониторинг группы для ментора

> **Контекст эволюции:** `conductor/architecture-evolution.md` (§2.3, §2.10). Прочитать перед началом.
> **Зависимости:** трек 1 (ЖЦ студента — статусы abandoned/advanced/not_advanced, abandonDetails, completionDetails), трек 2 (ContentPath — для истории шагов), трек 5 (прозрачность: S07/S08 доступны всем ролям).
> **Важно:** действия ментора в S08 (`mark-abandoned`, `complete-student`) уже реализованы в треке 1 (кнопки `⚠️ Неактивен`, `✅ Завершить`/`🔄 Сменить исход`). Этот трек **не создаёт их заново** — он добавляет bot-level тесты для `complete-student` UI (уже работает end-to-end после аудита Релиза 1), а также перерабатывает **контент** S07/S08.

## Обзор
Дать ментору (и любопытному — прозрачность) инструменты оперативного контроля группы: автоматическое выявление отстающих, сводка здоровья группы, сортировка проблемных сверху, детальная история шагов студента и динамика обучения.

**Роли:** S07/S08 контент — всем ролям (прозрачность из трека 5). Кнопки действий над студентом — только ментор потока / ADMIN.

## Функциональные требования

### F1. Логика «Отстаёт» (S07, S08)
- Статус «⚠️ Отстаёт» если:
  - `completedAt` последнего шага старше 5 дней (константа, configurable), статус `active`; ИЛИ
  - прогресс отстаёт от медианного по группе на 30%+.
- В S07 — эмодзи-маркер у студента (виден всем — прозрачность). В S08 — причина («нет активности 6 дней» / «отставание от группы 35%»).
- `abandonDetails: { who: 'mentor', cause: 'inactivity' }` — ментор может поставить статус через существующую кнопку `⚠️ Неактивен` (UC `mark-abandoned` из трека 1, cause=`inactivity`) прямо из карточки отстающего.

### F2. Сводка по группе (S07)
- Кроме «Всего: N»: медиана прогресса, число отстающих, число advanced, число not_advanced, число active, число enrolled.
- Видна всем ролям (прозрачность здоровья группы).

### F3. Сортировка списка студентов (S07)
- Отстающие вверху, затем по убыванию прогресса. Ментору важны проблемные первыми.

### F4. История шагов студента (S08 `history:*`)
- Реализовать обработчик `monitor:history:{studentId}` (сейчас заглушка «🚧 История шагов ещё не реализована»).
- Список шагов студента: позиция (ContentPath из трека 2), статус (issued/completed), даты выдачи/завершения.
- Виден всем (прозрачность — кто как учился). Ментор видит темп и где студент застрял.

### F5. Действия ментора в S08 — доводка и тесты (интеграция с треком 1)
Действия уже есть в коде (трек 1). Этот трек:
- `complete-student` UI уже починен (аудит Релиза 1): `complete-confirm` (диалог) и выполнение (`complete-confirm-confirm` → вызов UC с выбранным `outcome`) разведены. Треку нужны только bot-level тесты.
- **Добавить bot-level интеграционные тесты** через `BotRouter.handleCallback`: выбор исхода → confirm → выполнение → статус студента изменился, `−STUDENT`, TgFacade-сообщение (для advanced/not_advanced).
- Кнопка `⚠️ Неактивен` → `mark-abandoned` (cause: `inactivity`) — уже работает end-to-end (трек 1), добавить bot-level тест если нет.
- Кнопка `🔄 Сменить исход` (для advanced/not_advanced) → `complete-student` с новым исходом.

> ⚠️ В треке 1 кнопка называлась «Отчислить»/cause=`by_mentor` — это переименовано в «Неактивен»/cause=`inactivity`. В треке 6 cause остаётся `inactivity` (отсутствие активности). Не вводить `by_mentor` заново.

## Критерии приёмки
- [ ] Отстающие (5+ дней без активности / 30%+ от медианы) помечаются с причиной (видно всем).
- [ ] S07 показывает сводку здоровья группы (всем).
- [ ] Список отсортирован: отстающие сверху.
- [ ] История шагов студента работает (позиция, статус, даты).
- [ ] `complete-student` UI работает end-to-end через BotRouter (выбор исхода → confirm → UC → статус изменился, −STUDENT).
- [ ] Bot-level тесты действий ментора в `tests/bot`.
- [ ] Покрытие >80%, TDD.

## За рамками
- Broadcast/рассылка группе (бэклог, трек 7).
- Автоматическое отчисление по неактивности (только ручное `mark-abandoned`).
- Графики/аналитика во времени (динамика — текстом/числами, без графиков).
