# Спецификация: Регистрация студентов потока 1 через бот-команды

## Обзор
Две временные бот-команды для зачисления студентов в поток 1:
1. **`/register_student`** — индивидуальная самостоятельная регистрация студента
2. **`/register_inactive`** — массовая регистрация админом оставшихся студентов

Все файлы команд лежат в одной папке `student-registration/`, включая `README.md` с описанием и инструкцией по удалению.

## Функциональные требования

### FR-1: `/register_student` — индивидуальная регистрация

**Кто:** Любой пользователь бота

**Формат команды:** `/register_student [pN-lM]`
- Без аргумента — используется урок по умолчанию `p4-l1`
- С аргументом — указанный урок (например `/register_student p2-l3`)

**Сценарий:**
1. Студент пишет `/register_student [pN-lM]` в личку боту
2. Бот проверяет, что студент есть в хардкод-списке (по Telegram ID)
3. **Если нет в списке** — бот вызывает Telegram API `getChatMember` для группы `-1003960918937`:
   - Если пользователь в группе (status = member/administrator/creator):
     - Если его нет в системе → создаёт пользователя с ролью GUEST
     - Продолжает сценарий
   - Если пользователь НЕ в группе → отказ: «Ты не являешься участником группы потока»
4. Бот проверяет, что у студента нет активной записи Student в потоке 1. Если уже есть — сообщает об этом и показывает текущий шаг.
5. Бот парсит аргумент урока, находит первый шаг урока в contentSnapshot потока 1
6. Создаёт Student запись:
   - streamId = поток 1
   - userId = UUID студента
   - currentStepId = первый шаг указанного урока
   - status = active
   - steps: [{ stepId, status: 'issued', issuedAt: now }]
7. **Добавляет роль STUDENT пользователю** (через UserFacade.updateUserRole)
8. Сообщает: «Ты зарегистрирован в потоке! Твой текущий шаг: ...»

**Валидация ввода урока:**
- Формат: `pN-lM` где N и M — цифры
- Если формат неверный — сообщить об ошибке с примером правильного формата
- Если урок не найден в contentSnapshot — сообщить и предложить проверить номер

### FR-2: `/register_inactive` — массовая регистрация

**Кто:** Админ (роль ADMIN)

**Сценарий:**
1. Админ пишет `/register_inactive` в личку боту
2. Бот проверяет роль ADMIN. Если нет — отказ.
3. Бот получает список из 16 студентов (хардкод)
4. **Для каждого студента из списка**:
   - Проверяет существование пользователя по UUID
   - **Если пользователь не найден в системе**:
     - Вызывает Telegram API `getChatMember` для группы
     - Если пользователь в группе → создаёт пользователя как GUEST
     - Если не в группе → пропускает, считает ошибкой
   - Проверяет отсутствие активной записи Student в потоке 1
   - Создаёт Student запись с currentStepId = первый шаг p1-l5
   - **Добавляет роль STUDENT пользователю**
   - Считает успешные и пропущенные
5. Выводит отчёт: «Добавлено: X, Пропущено (уже есть): Y, Ошибки: Z»

**Обработка ошибок:**
- Если пользователь не найден в системе и не в группе — пропустить, посчитать как ошибку
- Если уже есть активная запись — пропустить, посчитать как «уже есть»
- Ошибка сохранения — посчитать как ошибку, продолжить со следующим

### FR-3: Роль STUDENT

После успешного зачисления студента в поток, обе команды **обязательно** добавляют пользователю роль `STUDENT` (если её ещё нет). Это гарантирует, что:
- Студент видит кнопку «Моя учёба» в главном меню
- Студент может проходить шаги через `LearningStory`

### FR-4: Временность и удаление

Обе команды временные. После использования:
1. Удалить папку `student-registration/` целиком
2. Удалить регистрацию сторис в `StreamController`
3. Удалить обработчики команд в `router.ts`
4. Откатить изменения в `ApiModule` (если были)

- [ ] После зачисления всегда добавляется роль STUDENT
- [ ] Student запись создаётся со всеми обязательными полями (steps, issuedAt, createdAt)
