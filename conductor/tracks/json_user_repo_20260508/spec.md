# Спецификация: Постоянное JSON-хранилище пользователей

## Обзор
Заменить inMemory-репозиторий пакета `user` на постоянное файловое JSON-хранилище. Данные пользователей сохраняются в один файл `users.json` в директории `data/users/` корня проекта. Предусмотреть общий адаптер `JsonFileRepo<T>` в пакете `core` как переиспользуемый базовый класс для будущих JSON-репозиториев (например, `course`).

## Функциональные требования

### FR1: JSON-хранилище пользователей
- Данные хранятся в одном файле `data/users/users.json` в виде JSON-массива объектов User
- Реализация интерфейса `UserRepo` из `packages/user/src/domain/user/repo.ts`
- Полная поддержка всех методов: `save`, `getByUuid`, `getByTelegramId`, `getAll` (с фильтрами), `isTelegramIdTaken`, `isEmpty`

### FR2: Общий JSON-адаптер (`JsonFileRepo<T>`)
- Пакет `core` получает базовый класс `JsonFileRepo<T>`, параметризованный:
  - Путём к файлу
  - Valibot-схемой для валидации
- Предоставляет универсальный механизм чтения/записи JSON-файла

### FR3: Валидация при чтении
- При загрузке данных из файла каждый объект проверяется через Valibot-схему `UserSchema`
- Невалидные записи пропускаются с предупреждением в лог (консоль)
- Валидные записи возвращаются как корректные данные

### FR4: Seed-данные
- Предусмотреть файл `data/users/seed.json` с начальными пользователями
- При первом запуске (файл `users.json` отсутствует) автоматически заполнить хранилище из seed
- Seed содержит минимум одного пользователя с ролью ADMIN

### FR5: Атомарность
- Блокировки не используются
- При сохранении — синхронная перезапись всего файла
- Конкурентный доступ не ожидается на текущем этапе

## Нефункциональные требования

### NFR1: Производительность
- Время чтения/записи для ~1000 пользователей не должно превышать 100 мс

### NFR2: Совместимость
- Существующий код, использующий `UserRepo` (команды, тесты), должен работать без изменений сигнатур

## Критерии приёмки
- [x] `UserJsonRepo` реализует все методы интерфейса `UserRepo`
- [x] Данные сохраняются в `data/users/users.json` и переживают перезапуск приложения
- [x] При отсутствии файла — автоматическое создание из seed
- [x] Битые данные в JSON не ломают приложение, а логируются и пропускаются
- [x] Все существующие тесты проходят с новой реализацией
- [x] `JsonFileRepo<T>` в core готов к использованию для course в будущем

## За рамками
- Миграция inMemory → JSON (стартуем с seed, а не переносим существующие inMemory-данные)
- Блокировки / конкурентный доступ
- Шифрование данных
- Бэкапы / версионирование файла
