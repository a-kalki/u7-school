# Спецификация трека Onboarding Domain Evolution

## 1. Обзор
Цель трека — отвязать прохождение анкеты (Questionnaire) от сущности User (заменить `userId` на `telegramId`), добавить поддержку промежуточных ответов (черновиков) в агрегат для работы со stateless ботом и создать единый Query UC для получения состояния анкеты.

## 2. Функциональные требования
- **Идентификация по Telegram ID**: Сущность `Questionnaire` должна использовать `telegramId` (число) вместо `userId`.
- **Поддержка черновиков**: Агрегат `QuestionnaireAr` должен хранить `draftAnswers: string[]` для текущего вопроса.
- **Новое действие**: Добавить команду `toggle-draft-answer` и метод `toggleDraftAnswer` в агрегат, чтобы добавлять/удалять выбранные варианты без перехода к следующему вопросу.
- **Обновление submit-answer**: Команда `submit-answer` для вопросов с множественным выбором должна использовать ответы из `draftAnswers` и затем очищать черновик.
- **Facade Query**: Создать `get-onboarding-state-uc`, возвращающий четкое состояние (`none_active` или `in_progress` с текущим вопросом и черновиками) по `telegramId`.
- **Роли пользователей**:
    - Модуль User должен предоставлять команду для гарантии наличия пользователя с ролью GUEST по telegramId.
    - После успешного завершения анкеты, пользователю должна выдаваться роль CANDIDATE.
- **Безопасность/Policies**: Убрать проверку `actorId` из OnboardingPolicies (команды анкеты публичны для телеграм-бота).

## 3. Критерии приёмки
- Схема `Questionnaire` обновлена, все тесты домена и API модуля `onboarding` проходят.
- В `QuestionnaireAr` корректно работают `toggleDraftAnswer` и валидация.
- Добавлены новые UseCases: `toggle-draft-answer-uc` и `get-onboarding-state-uc`.
- Интеграция с UserFacade работает корректно (добавление ролей GUEST/CANDIDATE).

## 4. За рамками
- Рефакторинг слоя бота (UI Controller) — это будет реализовано во втором треке.
