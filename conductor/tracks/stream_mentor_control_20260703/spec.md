# Спецификация трека: Управление потоком ментором

## Обзор

Трек добавляет 5 фич для расширения управления жизненным циклом потока со стороны ментора и улучшения процесса зачисления. Полная спецификация экранов — в `packages/stream/src/ui/bot/ui-spec.md` (экраны помечены 🔧).

| # | Фича | US | Экраны |
|---|------|----|--------|
| 1 | Проверочное слово (enrollmentKey) | US-10 | S09 (шаг 10), S10 |
| 2 | Отчисление студента | US-9 | S08 |
| 3 | Детали потока | US-11 | S02, S04 |
| 4 | Снятие CANDIDATE при зачислении | US-3 | S02 (логика) |

## Функциональные требования

### FR-1: Проверочное слово (enrollmentKey)

**Domain:**
- Новое поле `enrollmentKey?: string` в `StreamSchema`
- Новый опциональный параметр `enrollmentKey?` в `CreateStreamCmd`
- Новый опциональный параметр `enrollmentKey?` в `EnrollStudentCmd`

**API:**
- `create-stream-uc`: сохранять `enrollmentKey` при создании потока
- `enroll-student-uc`: если у потока задан `enrollmentKey` — сверять с переданным; при несовпадении — ошибка

**UI (story):**
- S09 шаг 10: ментор вводит слово или пропускает
- S10: новый экран — запрос слова у студента, 3 попытки, кнопка «Отмена»
- Архитектура: story проверяет для UX, use-case проверяет для безопасности (двойная проверка)

### FR-2: Отчисление студента

**Domain:**
- Новый статус `expelled` в `StudentSchema` (enum: `active`, `completed`, `expelled`)
- `StudentAr.expel()` — метод перевода в `expelled`
- `StudentPolicy.canExpel(actor)` — явный метод проверки прав (ментор потока или ADMIN)

**API:**
- Новый `expel-student-uc`: принимает `{ streamId, studentId }`
- Проверка: `StudentPolicy.canExpel(actor)`
- Действие: `StudentAr.expel()` → сохранить, `UserFacade.removeRole(userId, STUDENT)`
- Запись **не удаляется** из репозитория

**Infra:**
- `StreamApiModuleResolver`: добавить `userFacade: UserFacade`

**UI (story):**
- S08: кнопка `❌ Отчислить` в карточке студента
- Подтверждение: «⚠️ Вы уверены?» → `Да, отчислить` / `Отмена`

### FR-3: Детали потока

**UI (story):**
- S02: кнопка `📋 Детали` в карточке потока (на всех статусах)
- S04: новый экран — поля `goal`, `result`, `rules`, `targetAudience`, `additional`
- Кнопка `⬅️ Назад к потоку`

### FR-4: Снятие CANDIDATE при зачислении

**API:**
- `enroll-student-uc`: после `+STUDENT` — если у пользователя есть `CANDIDATE`, снять через `UserFacade.removeRole(userId, CANDIDATE)`

## Нефункциональные требования

- **NFR-1:** use-case'ы обращаются к другим модулям через фасад (`UserFacade`), получаемый из резолвера модуля
- **NFR-2:** story используют `appApi` для запросов к другим модулям
- **NFR-3:** `StudentAr` запись никогда не удаляется — только меняется статус
- **NFR-4:** Все новые use-case имеют unit-тесты, story — интеграционные тесты
- **NFR-5:** `bun run check:p stream` проходит без ошибок
- **NFR-6:** Интеграционные тесты в `tests/bot/integration/stream/` и e2e в `tests/bot/e2e/stream/`

---

## Критерии приёмки

- [ ] Ментор может задать кодовое слово при создании потока; пустое = свободная запись
- [ ] Студент вводит слово (3 попытки), при успехе — зачисление, при провале — возврат
- [ ] Use-case `enroll-student` отклоняет запись с неверным `enrollmentKey`
- [ ] Ментор может отчислить студента: кнопка → подтверждение → статус `expelled` + `-STUDENT`
- [ ] При зачислении снимается `CANDIDATE` (если был)
- [ ] Кнопка «Детали» показывает расширенные поля на всех статусах
- [ ] Все существующие тесты продолжают проходить

## За рамками

- Снятие ролей при завершении потока — бэклог (требует анализа: кто может менять роль — ментор, админ, сам пользователь)
- Сертификация студентов при завершении (`completed_and_passed`) — бэклог
- Самостоятельный выход из потока — бэклог
- Заморозка студента — бэклог
