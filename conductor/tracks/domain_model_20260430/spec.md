# Specification: Разработка доменной модели платформы (модуль core)

## Overview
Данный трек направлен на создание базовой доменной модели платформы **u7-school** в виде отдельного модуля `packages/core`. Модель будет описывать структуру образовательного контента (курсы, модули, уроки) и пользователей системы, обеспечивая строгую валидацию данных на этапе выполнения.

## Functional Requirements

### 1. Модель Пользователя (User)
- Поддержка ролей: `STUDENT`, `MENTOR`, `ADMIN`.
- Базовые поля: ID, имя, telegramId, роль.

### 2. Структура Курса (Course) — проектно-ориентированный подход
- **Курс (Course):** Корневой элемент. Содержит метаданные (название, описание, автор) и *либо* список **Модулей**, *либо* список **Проектов** (исключающее ИЛИ). Ментор выбирает один из подходов при создании курса.
    * автор: ссылка на автора;
    * для кого: описание для кого предназначен курс;
    * цель: какую цель преследует курс;
    * результат: что будет уметь студент в конце курса;
    * правила: различные правила прохождения курсов;
    * дополнительно: поле куда ментор может добавить дополнительную инфу;
- **Модуль (Module):** Логическая единица курса. Содержит список **Проектов**.
    * цель: цель модуля;
    * результат: что будет уметь, или какой результат получит студент;
    * дополнительно: дополнительная инфа;
- **Проект (Project):** Основная рабочая единица. Содержит список **Уроков**.
    * цель: цель проекта;
    * результат: что будет уметь, или какой результат получит студент;
    * дополнительно: дополнительная инфа;
- **Урок (Lesson):** Изучение определенной темы, состоит из шагов:
    *   Название.
    *   Дополнительно.
    * **Шаг (Step):** Атомарное задание для студента. Абстрактное и расширяется через наследование:
      - Описание: что нужно сделать.
    * **Практика с ментором:** Список тем, ментор может для закрепления добавить дополнительные материалы (например: показать работу интерпретатора в режиме repl; разобрать файл с практическими примерами разных случаев темы ветвления). Тоже абстрактное и расширяется через наследование. Нужно явно определить доменное имя и имя объекта с условным названием список тем.

### 3. Валидация и Типизация
- Использование **Valibot** для создания схем валидации (Runtime validation).
- Автоматическая генерация TypeScript типов из схем Valibot.

## Non-Functional Requirements
- **Расположение:** Новый пакет `packages/core` в монорепозитории.
- **Структура:** Пакет имеет два слоя (папки):
  - domain: доменные объекты;
  - api: слой приложения;
- **Стандарт кода:** Соответствие Biome (линтер/форматтер).
- **Тестирование:** Покрытие TDD согласно workflow проекта.

## Acceptance Criteria
- [ ] Определены Valibot-схемы для всех сущностей (User, Course, Module, Project, Lesson, Step, File).
- [ ] Экспортированы соответствующие TypeScript типы.
- [ ] Реализованы юнит-тесты, проверяющие валидность и невалидность данных для каждой схемы.
- [ ] Модуль `packages/core` корректно инициализирован и доступен для импорта.

## Out of Scope
- Реализация логики прохождения курсов (Enrollments/Progress) — будет в следующих треках.
- Реализация API эндпоинтов (Elysia) или логики бота (Grammy).
- Хранение данных в БД (будет реализовано в слое адаптеров/репозиториев позже).
