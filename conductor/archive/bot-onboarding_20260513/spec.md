## Обзор
Реализовать Telegram-флоу onboarding в `apps/u7-bot` через grammY + $T1

1. bot+conversations это высокоуровневый контекст, он в будущем возможно будет "помнить или знать" что пользователь находится на стадии "онбординга" или "прохождения курсов" и запускать соответсвующие ветки взаимодействия;
1. controller это прокладка между bot и apiApp. Он знает что он предоставляет ux через бота для модуля onboarding и забирает на себя всю техническую работу в связи с этим. Также может предоставить прямой доступ к apiApp, если бот захочет напрямую дергать usecase-ы;
1. apiApp, точка входа в api, он предоставляет доступ (api) чтобы внешний клиент мог дергать вложенные в него модули и usecases;
1. в системе бот делает запросы как ADMIN пользователь. UUID бота-админа передаётся через env (`BOT_ADMIN_UUID`). Авторегистрация не предусмотрена — администратор должен предварительно создать пользователя-бота через CLI/API и указать его UUID в конфигурации;
1. В запросах в onboarding в actor ставится бот. Если нужно передать пользователя — передавай его в команде к uc;

## Функциональные требования
### /start
- Пользователь впервый нажимает `/start`.
- Приходит приветствие с inline-клавиатурой: «Быть в курсе» / «Стать студентом».
- Для каждой кнопки пояснение что означает (какие профиты) это действие.

### Логика повторного /start
Роли пользователя могут комбинироваться (например `[GUEST, SUBSCRIBER, CANDIDATE]`). Роль `GUEST` никогда не удаляется — она остаётся навсегда как базовая. Роль `SUBSCRIBER` добавляется при вступлении в группу (за рамками трека), `CANDIDATE` — при завершении анкеты.

Приоритет при определении flow:
- **Есть роль CANDIDATE** (вне зависимости от SUBSCRIBER) → предложить заполнить новую заявку. Создаётся новая анкета, история сохраняется.
- **Есть роль SUBSCRIBER, но нет CANDIDATE** → короткое сообщение «Ты уже с нами!».
- **Только GUEST** (нет CANDIDATE и SUBSCRIBER) → показать меню с «Быть в курсе» / «Стать студентом».
- **Новый пользователь** → приветствие + меню.

### «Быть в курсе»
- Показать ссылку на новостную группу (из `.env`).
- Роль SUBSCRIBER выдаётся при вступлении в группу — за рамками этого трека.
- Кнопка «Вернуться в меню».

### «Стать студентом» — Conversation
1. Приветствие, объяснение анкеты.
2. Пошаговый опросник с inline-клавиатурами:
   - Множественный выбор: кнопки обновляются с ✅, кнопка «Далее ➡️».
   - Одиночный выбор: сразу переход к следующему вопросу.
   - Ветвление: после `intensity` показывать `baseDays`+`baseTime` ИЛИ `intensiveTime`.
3. После последнего ответа анкета автоматически завершается (`submit-answer` переводит статус в `completed`), пользователю выдаётся роль `CANDIDATE`.
4. Сообщение-confirmation + ссылка на группу.

### /cancel
- Прерывает текущую conversation.
- Удаляет промежуточные данные из session.
- Возвращает в главное меню.

### Дополнительно
- Конфигурация через `.env` (Bun загружает автоматически, API dotenv не требуется).

## Нефункциональные требования

- Session storage в JSON (восстановление состояния при перезапуске), если нужно используй JsonRepo чтобы был удобный api.
- Покрытие тестами где возможно (моки grammY).

## Критерии приёмки
- [ ] Бот отвечает на `/start`, создаёт/находит пользователя.
- [ ] Inline-меню работает корректно.
- [ ] Conversation проходит все вопросы анкеты.
- [ ] Множественный выбор и ветвление работают корректно.
- [ ] Заявка создаётся в БД, пользователь получает CANDIDATE.
- [ ] `/cancel` прерывает опросник и очищает session.
- [ ] Повторный `/start` корректно обрабатывает существующих кандидатов и подписчиков.
- [ ] Конфигурация через `.env`.

## За рамками
- Выдача роли SUBSCRIBER при вступлении в группу (отдельный процесс).
- Розыгрыш/распределение по потокам.
- Admin-меню для просмотра заявок (будущий трек).
