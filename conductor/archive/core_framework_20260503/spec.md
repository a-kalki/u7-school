# Трек: Абстрактный каркас приложения (@u7/core)

## Обзор
Создать новый пакет `@u7/core` (`packages/core/`) — абстрактный каркас приложения.
Определяет базовые классы и контракты для модулей, use-case'ов (uc), агрегатов (ar),
обработки ошибок. Доменные модули будут использовать его как каркас. Core определяет свои классы разделив по структуре:
```
```
- shared: общая логика и объекты для фронта и бэка
- domain: доменные оъекты и логика
  - shared: общая логика домена
  - ...: другие папки
- api: логика и объекты для бэка
  - shared: общая логика для бэка
  - ...: другие папки
- ui: логика и объекты для фронта
  - shared: общая логика для фронта
  - ...: другие папки
- infra: возможнная папка логики и объектов инфрастукруры
```
```

## Функциональные требования

### FR1: Объекты ошибок и исключения
- Ошибки делятся на уровни: `"domain"` и `"api"`
- Плоские объекты ошибок: просмотреть текущие объекты исключений ApiException, DomainException, реализовать объекты ошибок
- Агрегат и use-case объявляют возможные ошибки в дженерике Meta
- Встроенный хелпер в абстрактные uc, ar `throwError(error)` — принимает объект ошибки, выбрасывает исключение (можно предложить другую реализацию ведущую к достижению цели: надежным контрактам)
- Исключение ловится на уровне контроллера и обрабатывается здравым способом
- **Цель:** фронт и бэк общаются явными объектами ошибок; линтер видит контракт; программист просто выбрасывает ошибку, не возвращая

### FR2: UC Meta — контракт use-case
Дженерик `UseCaseMeta` определяет:
- `commandName` — имя команды, которое use-case обрабатывает
- `input` — входящие данные (команда)
- `output` — исходящие данные (результат)
- `errors` — все возможные ошибки (включая ошибки агрегатов, которые use-case вызывает)

UC Meta точно определяет только те ошибки, которые будут в ответе use-case, не перегружая всеми возможными ошибками агрегата.

### FR3: Агрегат — контракт ошибок
- Агрегат имеет свой дженерик Meta с пулом возможных ошибок
- Методы агрегата через встроенный хелпер выбрасывают ошибки из объявленного пула и в идеале объявляют в контрактах метода чтобы uc понимал какие ошибки ожидать. Система типов следит за этими контрактами на уровнях ar, uc
- UC Meta может включать ошибки метода агрегата в свой контракт (вкладывать)
- Если агрегат выбрасывает ошибку из своего пула, она соответствует контракту use-case и не требует перехвата

### FR4: Абстрактный класс UseCase
- `execute(command, actorId?)` — основной метод
- Свойства: схема валидации (Valibot), имя команды которую она обрабатывает, возможно свойство которая определяет для доступна ли выполнение неавторизованным пользователям (чтобы не проверять его в каждом дочернем uc)
- Все логики обработки свойств из верхнего пункта определены в базовом классе, но могут быть переопределены или расширены дочерними uc
- Максимум ООП: конкретные use-case сконцентрированы на доменной логике, базовая логика у родителя
- имеет доступ к резолверу модуля через дженерик (тип) и свойство (значение) чтобы динамически получить нужные зависимости (например репозитории)

### FR5: Абстрактный класс Module
- `useCases: UseCase[]` — массив use-case'ов для модуля
- `name`: имя модуля, используется как url модуля
- `handle(moduleCommand)` — принимает запрос извне
  - moduleCommand: {name, attrs (или command), actorId?}
  - Находит use-case по имени команды
  - Передаёт `attrs` и `actorId`
  - Возвращает результат
- Дженерик `TResolve` — плоский объект-резолвер, передаётся в конструкторе или `init()`
- Модуль передаёт резолвер всем дочерним use-case

### FR6: CLI-контроллер
- Создаёт `ModuleCommand` из CLI-аргументов
- Передаёт модулю
- Ловит исключения, восстанавливает объекты ошибок
- Формирует JSON-ответ (успех) или JSON-ошибку (провал)
- Наследует/переиспользует логику из абстрактного каркаса

### FR7: Поддержка OpenAPI-вывода (опционально, задел)
- Предусмотреть в архитектуре модуля предоставление его api, скорее всего выдает массив json-схем для openApi, возможно в структурированном виде (группировка по агрегатам)
- Можно запросить мета-информацию: какие команды принимает модуль, их схемы
- Реализовать базовую поддержку в абстрактном классе модуля или если это ресурсоемко, то оставить это на будущие треки

## Нефункциональные требования
- Покрытие тестов >80%
- Соответствие Google TypeScript Style Guide (Biome)
- Полная типобезопасность — дженерики Meta проверяются на уровне типов
- `@u7/course` должен быть обновлён для использования нового каркаса

## Критерии приёмки
- [ ] Пакет `@u7/core` создан, собирается, тесты проходят
- [ ] Определены плоские объекты ошибок, хелпер `throwError`, исключения
- [ ] `UseCaseMeta<T>` — дженерик с input, output, errors, commandName
- [ ] `UseCase<TMeta, TResolve>` — абстрактный класс с `execute()` и валидацией
- [ ] `AggregateMeta<T>` — дженерик с мета (имя, пул ошибок), хелпер выбрасывания
- [ ] `Module<TResolve>` — абстрактный класс с `handle()` и `useCases`
- [ ] CLI-контроллер работает с новым каркасом
- [ ] CLI-контроллер обрабатывает ошибки: ловит исключения → JSON-ошибка
