# Резюме руководства по стилю TypeScript от Google

Этот документ обобщает основные правила и рекомендации из руководства по стилю TypeScript от Google, которые обеспечиваются инструментом `gts`.

## 1. Особенности языка
- **Объявление переменных:** Всегда используйте `const` или `let`. **`var` запрещен.** По умолчанию используйте `const`.
- **Модули:** Используйте модули ES6 (`import`/`export`). **Не используйте `namespace`.**
- **Экспорт:** Используйте именованный экспорт (`export {MyClass};`). **Не используйте экспорт по умолчанию (default export).**
- **Классы:**
  - **Не используйте приватные поля `#private`.** Используйте модификатор видимости `private` в TypeScript.
  - Помечайте свойства, которые никогда не переназначаются вне конструктора, как `readonly`.
  - **Никогда не используйте модификатор `public`** (он является значением по умолчанию). Ограничивайте видимость с помощью `private` или `protected`, где это возможно.
- **Функции:** Предпочитайте объявление функций (function declarations) для именованных функций. Используйте стрелочные функции для анонимных функций/обратных вызовов.
- **Строковые литералы:** Используйте одинарные кавычки (`'`). Используйте шаблонные литералы (`` ` ``) для интерполяции и многострочных строк.
- **Проверки на равенство:** Всегда используйте строгое равенство (`===`) и неравенство (`!==`).
- **Утверждение типов (Type Assertions):** **Избегайте утверждений типов (`x as SomeType`) и утверждений о ненулевом значении (`y!`)**. Если вы должны их использовать, предоставьте четкое обоснование.

## 2. Запрещенные возможности
- **Тип `any`:** **Избегайте `any`**. Предпочитайте `unknown` или более конкретный тип.
- **Объекты-обертки:** Не создавайте экземпляры классов-оберток `String`, `Boolean` или `Number`.
- **Автоматическая вставка точки с запятой (ASI):** Не полагайтесь на нее. **Явно завершайте все инструкции точкой с запятой.**
- **`const enum`:** Не используйте `const enum`. Вместо этого используйте обычный `enum`.
- **`eval()` и `Function(...string)`:** Запрещены.

## 3. Именование
- **`UpperCamelCase`:** Для классов, интерфейсов, типов, перечислений (enums) и декораторов.
- **`lowerCamelCase`:** Для переменных, параметров, функций, методов и свойств.
- **`CONSTANT_CASE`:** Для глобальных константных значений, включая значения перечислений.
- **Префикс/суффикс `_`:** **Не используйте `_` в качестве префикса или суффикса** для идентификаторов, в том числе для приватных свойств.

## 4. Система типов
- **Вывод типов (Type Inference):** Полагайтесь на вывод типов для простых, очевидных типов. Будьте явными для сложных типов.
- **`undefined` и `null`:** Оба поддерживаются. Будьте последовательны в рамках своего проекта.
- **Необязательные параметры vs `|undefined`:** Предпочитайте необязательные параметры и поля (`?`) добавлению `|undefined` к типу.
- **Тип `Array<T>`:** Используйте `T[]` для простых типов. Используйте `Array<T>` для более сложных объединений типов (например, `Array<string | number>`).
- **Тип `{}`:** **Не используйте `{}`**. Предпочитайте `unknown`, `Record<string, unknown>` или `object`.

## 5. Комментарии и документация
- **JSDoc:** Используйте `/** JSDoc */` для документации, `//` для комментариев к реализации.
- **Избыточность:** **Не объявляйте типы в блоках `@param` или `@return`** (например, `/** @param {string} user */`). Это избыточно в TypeScript.
- **Добавление информации:** Комментарии должны добавлять информацию, а не просто пересказывать код.

*Источник: [Google TypeScript Style Guide](https://google.github.io/styleguide/tsguide.html)*
