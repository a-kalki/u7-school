# Политика прав (Policy) — Styleguide

**Назначение:** файл `domain/<entity>/policy.ts` определяет правила доступа к операциям над сущностью. Stateless — синхронно проверяет права на основе ролей актора.

---

## 1. Ключевые правила

1. Экспортирует plain-object или набор функций (`UserPolicy.canCreate(actor)`).
2. **Только синхронные проверки** — без `async`, без обращений к репозиторию/БД.
3. Каждый метод принимает `actor` и опционально `target`, возвращает `boolean`.
4. Используется на слоях api/ui **перед** выполнением бизнес-логики, не внутри агрегата.
5. Проверяйте роли через enum `Role`, не магические строки.
6. **Логика доменных прав НЕ должна утекать из своего модуля.** Проверки структуры сущности (фазы, связи, порядок) — в политике пакета-владельца. Проверки статусов/свойств чужой сущности — в политике её пакета. Если политика одного модуля нуждается в проверках из другого — она **делегирует** вызов (импортирует и вызывает чужую Policy), а не дублирует логику у себя. Пример: `StreamPolicy.canEnrollNextModule` делегирует проверки структуры курса в `CoursePolicy.containsModule` / `CoursePolicy.isFirstModule`, оставляя у себя только проверку статуса студента.

Живой пример: [`packages/stream/src/domain/stream/policy.ts`](../../../packages/stream/src/domain/stream/policy.ts).

```typescript
export const StreamPolicy = {
	canCreate(actor: User): boolean {
		return actor.roles.includes(Role.MENTOR);
	},
	canEdit(actor: User, target: Stream): boolean {
		return actor.roles.includes(Role.ADMIN) || actor.uuid === target.mentorId;
	},
};
```

---

## 2. Тестирование

- Группируйте тесты по методам на втором уровне (`describe("canCreate")`).
- Покрывайте все роли (`ADMIN`, `MENTOR`, `STUDENT`) и случаи: доступ разрешён, доступ запрещён, доступ к чужому объекту, доступ к своему.
- Не выходите за пределы ответственности объекта.

Живой пример теста: `packages/stream/src/domain/stream/policy.test.ts`.

---

## Регресс

**Недопустимо** ломать существующий функционал и тесты, не связанные с текущей задачей: меняй только то, что относится к задаче, не удаляй и не переписывай корректные тесты — только добавляй новые. Подробные правила — [testing.md, §«Защита от регресса и чистота правок»](../testing.md).
