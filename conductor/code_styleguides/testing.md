# Правила тестирования

Проект следует принципу **TDD (Test-Driven Development)** и стремится к высокому покрытию кода (>80%).

## Изоляция тестов

- Тесты **не должны зависеть** друг от друга. Каждый тест создает свое состояние с нуля.
- Используйте `beforeEach` или создавайте свежие экземпляры зависимостей внутри каждого теста.
- Изоляция гарантирует, что порядок запуска тестов не влияет на результат.

## Размер и читаемость

- Целевой размер одного теста: **10–20 строк**.
- Если тест становится слишком длинным, выносите настройку данных в хелперы или `beforeEach`.
- Тесты должны быть максимально простыми для чтения — это документация поведения кода.

## Проверка исключений (AppException)

При тестировании ошибочных сценариев запрещено проверять только класс исключения. Тест **обязан** проверять конкретную ошибку.

```typescript
// Плохо: проверяем только класс
expect(() => uc.execute(...)).toThrow(AppException);

// Хорошо: проверяем имя ошибки или сообщение
try {
  await uc.handle(...);
} catch (e: any) {
  expect(e.error.name).toBe("USER_NOT_FOUND");
  expect(e.error.kind).toBe("not-found");
}
```

## Тестирование агрегатов vs схем

1. **Схемы валидации:** На них пишутся детальные тесты на все поля, граничные случаи и типы данных.
2. **Агрегаты:** Достаточно одного-двух тестов, подтверждающих, что валидация инвариантов (схема) подключена. Основные тесты агрегата фокусируются на **бизнес-логике** (методы изменения состояния, сложные переходы).

## Инструментарий

- Тест-раннер: **Bun Test**.
- Запуск всех тестов: `bun test`.
- Запуск с покрытием: `bun test --coverage`.
