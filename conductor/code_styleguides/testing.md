# Правила тестирования

Проект следует принципу **TDD (Test-Driven Development)** и стремится к высокому покрытию кода (>80%).

## Изоляция тестов

- Тесты **не должны зависеть** друг от друга. Каждый тест создает свое состояние с нуля.
- Используйте `beforeEach` или создавайте свежие экземпляры зависимостей внутри каждого теста.
- Изоляция гарантирует, что порядок запуска тестов не влияет на результат.

## Размер и читаемость

- Целевой размер одного теста: **10–20 строк**.
- Если тест становится слишком длинным, выносите настройку данных в хелперы или `beforeEach`.
- Тесты должны быть максимально простыми для чтения — это документация поведения кода.

## Проверка исключений (AppException)

При тестировании ошибочных сценариев запрещено проверять только класс исключения. Тест **обязан** проверять конкретную ошибку.

```typescript
// Плохо: проверяем только класс
expect(() => uc.handle(...)).toThrow(AppException);

// Хорошо: проверяем имя ошибки или сообщение
expect(() => uc.handle(...)).toThrow('текст ошибки');

// Тоже хорошо
try {
  await uc.handle(...);
  expect('this').toBe('not be call');
} catch (e: AppException) {
  expect(e).toInstanceOf(AppException);
  expect(e.error.name).toBe("USER_NOT_FOUND");
  expect(e.error.kind).toBe("not-found");
}
```

## Тестирование агрегатов vs схем

1. **Схемы валидации:** На них пишутся детальные тесты на все поля, граничные случаи и типы данных.
2. **Агрегаты:** Достаточно одного-двух тестов, подтверждающих, что валидация инвариантов (схема) подключена. Основные тесты агрегата фокусируются на **бизнес-логике** (методы изменения состояния, сложные переходы).

## Пирамида тестов

Для обеспечения качества используется многоуровневый подход:

### API (Бэкенд)
1. **Домен (Domain):** Максимальное покрытие. Тщательное тестирование всех бизнес-правил, инвариантов и пограничных случаев внутри агрегатов и схем.
2. **Репозитории (Repo):** Тестирование контрактов и реализации (включая In-memory) для обеспечения гарантированно безошибочной работы с данными.
3. **Use Case (UC):** Проверка сценариев использования. Тестируются цепочки: ограничение доступа по авторизации -> ошибка, отсутствие сущности -> not-found, нарушение прав -> access-denied, успех -> сохранение в Repo.
4. **Модуль (Module):** Дымовое тестирование. Обычно достаточно одного успешного теста на каждый Use Case, чтобы убедиться, что он правильно подключен в диспетчере модуля.

### UI (Фронтенд)
- **E2E Тесты:** Выполняются для каждой пользовательской истории (`User Story` / `UseCase`) с внешнего периметра. Основной фокус на удачных сценариях ("Happy Path"). Негативные сценарии добавляются по необходимости при высокой критичности.
- **MarkdownV2-валидация:** Каждый `BotResponse` в тестах стори, контроллеров и e2e **обязан** проходить `assertResponseMarkdownSafe(response)` из `@u7-scl/core/ui`. См. [MarkdownV2 в Telegram-боте](markdown-bot.md).

## Защита от регресса и чистота правок

**КРИТИЧНО:** Новая реализация, исправление багов или рефакторинг не должны приводить к регрессу функционала, не относящегося к текущей задаче.

- **Запрещено:** Вносить изменения в функции или тесты, которые не входят в область текущей задачи.
- **Исключение:** Если правка соседнего кода абсолютно необходима (например если вы заметили критическую ошибку или для выполнения текущей задачи), вы **обязаны** сообщить об этом пользователю в чате и получить явное разрешение на редактирование.
