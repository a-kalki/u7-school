# Общая архитектура приложения

## Обзор

Платформа **u7-school** построена по модульному принципу. Приложение разделено на доменные модули, каждый из которых инкапсулирует свою предметную область. Взаимодействие с модулями происходит через команды (commands), что позволяет единообразно подключать разные интерфейсы: консоль, Telegram-бота и REST API.

## Доменные модули

Приложение делится на независимые доменные модули (course, w3school и др.). Каждый модуль — это отдельный npm-пакет в директории `packages/`.

## Многослойная архитектура модуля

Каждый модуль внутри разделен на три слоя:

```
packages/<module>/
  src/
    domain/       # Слой предметной области (чистый код, без инфраструктуры)
    api/          # Слой приложения (оркестрация, репозитории, use cases)
    ui/           # Слой интерфейса пользователя (фронтенд-компоненты, если применимо)
```

### Слой api: Структура папок

```
src/api/
  <command-name>-uc.ts   # Сценарии использования (Use Cases)
  module.ts              # Entry point модуля
```

## Вход в модуль: Команды и Entry Point

Каждый модуль предоставляет единый **entry point** — класс `Module`, с методом `execute(uc-name: sting, ...other)` выполняющий указанный usecase (uc):

### Поток выполнения команды

1. контроллер (или story для bot,cli сценариев) ловит внешний триггер и формирует вызов module.execute(). В некоторых случаях возможен вызов через объект приложения apiApp.execute(), предоставляющий вызов ко всем модулям приложения, который в свою очередь делегирует вызов соответствующему модулю.
2. **Entry point модуля `execute(...)`:** Диспетчер команд модуля находит нужный Use Case.
3. **Use Case:**
    - Проверка валидации команды (через `inputSchema`).
    - Проверка прав (через `Policy`).
    - Изменение состояния через агрегат (`Ar`).
    - Сохранение через репозиторий (`Repo`).
4. **Result:** Возврат результата вызывающей стороне.

## Layer-based импорты

### Схема слоёв и направление зависимостей

Каждый пакет предоставляет точки входа по слоям через `exports` в `package.json`. Внешние потребители импортируют только через слойные точки входа:

```
@u7/core/domain    — чистый домен (Aggregate, AppError, фабрики ошибок)
@u7/core/api       — сценарии использования (UseCase, Module)
@u7/core/ui        — контроллеры (для bot, cli)
@u7/core/shared    — общий код (isoNow)
```

**Направление зависимостей:**

```
shared → domain → (api, ui)
```

- `domain` **не импортирует** `api` или `ui`.
- `api` может импортировать `domain`.
- `ui/bot/controller/` и `cli-controller` могут импортировать `api`.
- `ui/web` (браузерный) **не импортирует** `api` напрямую.

### Настройка package.json

Каждый пакет должен иметь поля `exports` (для внешних импортов) и `imports` (для внутренних `#`-импортов):

```jsonc
// packages/<module>/package.json
{
  "exports": {
    "./domain": "./src/domain/index.ts",
    "./api": "./src/api/index.ts",
    "./ui": "./src/ui/index.ts",
    "./infra": "./src/infra/index.ts"
  },
  "imports": {
    "#domain/*": "./src/domain/*.ts",
    "#api/*": "./src/api/*.ts",
    "#ui/*": "./src/ui/*.ts",
    "#infra/*": "./src/infra/*.ts"
  }
}
```

### Слойные index.ts (barrel exports)

Каждый слой должен иметь `index.ts`, ре-экспортирующий все публичные классы и типы. Например `src/domain/index.ts`:

```typescript
export { Aggregate } from "./ar/aggregate";
export { errNotFound, throwError } from "./errors/error-helpers";
export type { AppError, NotFoundError } from "./errors/errors";
```

### Примеры импортов

**Внешние (из другого пакета):**
```typescript
import { Aggregate } from "@u7/core/domain";
import { UseCase } from "@u7/core/api";
import { UserAr } from "@u7/user/domain";
```

**Внутренние (внутри пакета, через `#`):**
```typescript
import { throwError } from "#domain/errors/error-helpers";
import type { UseCase } from "#api/uc/use-case";
import { UserUseCase } from "#api/user-uc";
```

### Flat-импорты запрещены

Импорты без суффикса слоя (`@u7/core`, `@u7/user`) **запрещены**. Каждый импорт должен явно указывать слой-источник.

### TypeScript paths

В корневом `tsconfig.json` должны быть прописаны пути для каждого слойного экспорта:

```json
{
  "paths": {
    "@u7/core/domain": ["packages/core/src/domain/index.ts"],
    "@u7/core/api": ["packages/core/src/api/index.ts"],
    "@u7/core/ui": ["packages/core/src/ui/index.ts"],
    "@u7/core/shared": ["packages/core/src/shared/index.ts"]
  }
}
```

## Межмодульное взаимодействие и DI

Проект следует строгой модульной архитектуре: прямое связывание между модулями минимизировано для независимости и тестируемости.

### Инверсия зависимостей (DI)

Каждый модуль при инициализации принимает объект зависимостей — **`Resolve`** (например, `CoreResolve`). Тип `Resolve` явно определён внутри модуля, чтобы клиенты знали, какие репозитории/сервисы требуются. Модуль передаёт этот объект в Use Cases через `init()`. В тестах вместо реальных зависимостей передаются моки или In-memory реализации.

См. живой код: `packages/core/src/api/module/module.ts` (`CoreResolve`).

### Правила взаимодействия между модулями

1. **Изменение и чтение состояния** — только через публичный API модуля (метод `execute()` или фасад).
2. **Запрет прямого импорта Use Case:** Прямой импорт и вызов `UseCase` другого модуля **запрещён**. Все действия — через команды модуля.
3. **Обмен доменными объектами:** Допускается прямой импорт доменных сущностей, схем, агрегатов и политик из другого модуля, если соблюдена иерархия зависимостей пакетов.

### Фасады (Facade)

Для упрощения доступа к часто используемым данным модуля (например, текущий пользователь) создаются фасады, скрывающие за собой вызовы команд. См. [facade.md](./skills/facade.md).
