# Межмодульное взаимодействие и DI

Проект следует строгой модульной архитектуре. Прямое связывание между модулями минимизировано для обеспечения независимости и тестируемости.

## Инверсия зависимостей (DI)

Каждый модуль при инициализации принимает объект зависимостей — **`Resolve`**.

- Тип `Resolve` должен быть явно определен внутри модуля (например, `CoreResolve`), чтобы клиенты точно знали, какие зависимости (репозитории, сервисы) требуются.
- Модуль передает этот объект своим Use Cases через метод `init()`.
- В тестах вместо реальных зависимостей (БД, API) в `Resolve` передаются моки или In-memory реализации.

```typescript
// Пример интерфейса резолвера в packages/core/src/api/module/module.ts
export type CoreResolve = {
  userRepo: UserRepository;
  courseRepo: CourseRepository;
}
```

## Правила взаимодействия между модулями

1. **Изменение и чтение состояния:** Только через публичный API модуля (метод `handle()` или фасад). 
2. **Запрет на импорт Use Case:** Прямой импорт и вызов `UseCase` другого модуля **запрещен**. Все действия выполняются через команды модуля.
3. **Обмен доменными объектами:** Допускается прямой импорт доменных сущностей, схем, агрегатов и политик из другого модуля, если соблюдена иерархия зависимостей пакетов.

## Фасады (Facade)

Для упрощения доступа к наиболее часто используемым данным модуля (например, получение текущего пользователя) могут создаваться фасады, скрывающие за собой вызовы команд.
