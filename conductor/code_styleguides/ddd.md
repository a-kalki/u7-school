# Принципы DDD (Domain-Driven Design)

Слой `src/domain/` содержит чистый код предметной области: типы, схемы валидации, агрегаты и политики.

## Основные концепции

### Типы (Entity)
Структуры данных предметной области. Обычно выводятся напрямую из Valibot-схем для обеспечения синхронизации типов и правил валидации.

### Схемы (Schema)
Valibot-схемы для валидации инвариантов и структур данных. Схемы сущностей являются "единственным источником истины" для всех DTO (команд, запросов).

### Агрегаты (Ar)
Классы, инкапсулирующие состояние и поведение (богатая доменная модель).
- Принимают состояние через конструктор.
- Предоставляют методы для изменения состояния.
- **Инварианты:** Проверяются в конструкторе и фабричных методах (например, `create()`). Это гарантирует, что агрегат всегда находится в валидном состоянии.
- Наследуются от базового класса `Aggregate<TMeta>`.

### Политики (Policy)
Объекты или функции, проверяющие права доступа (разрешения на действия). Используются в Use Cases для проверки прав актора перед выполнением бизнес-логики.

## Правила валидации в DTO

Все команды (`Command`) и другие объекты передачи данных, зависящие от доменных сущностей, **обязаны** наследовать правила валидации из соответствующих схем сущностей.

**Пример:**
```typescript
import * as v from "valibot";
import { UserSchema } from "../../domain/user/user";

export const CreateUserCommandSchema = v.object({
  // Наследование правила для поля name
  name: UserSchema.entries.name, 
  role: UserSchema.entries.role,
});
```
Это исключает дублирование логики валидации и гарантирует консистентность данных во всей системе.
