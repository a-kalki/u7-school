# Инструкция по наполнению уроков шагами

## Нумерация уроков и папок

Все папки уроков в `data/lessons/nur/` именуются по строгому шаблону:

```
p<номер_проекта>-l<номер_урока_в_проекте>-<kebab-case-название>
```

- `p` — номер проекта (1, 2, 3...)
- `l` — номер урока **внутри** проекта (1, 2, 3...)
- Название — короткое описание темы на английском, kebab-case

**Примеры:**
- `p1-l1-introduction-to-programming`
- `p1-l5-data-types-exceptions`
- `p2-l3-data-structures-array-set`
- `p3-l4-functions-type-properties-styles`
- `p4-l5-recursion`

### Правила нумерации

1. Каждый проект имеет свой счётчик уроков, начинающийся с 1.
2. Порядок уроков внутри проекта определяется порядком `lessonIds` в `data/courses/modules.json`.
3. Сквозная нумерация (01, 02… 28) не используется — только `pN-lM`.

### Видео-ссылки в контенте

В поле `content` шагов (`steps.json`) и в тексте `lesson.md` видео-ссылки оформляются с новым номером:

```
[p<проект>-l<урок>. Название урока.mp4](https://drive.google.com/...)
```

✅ Правильно: `[p1-l1. Введение в программирование.mp4](...)`
❌ Неправильно: `[01. Введение в программирование.mp4](...)`

### Создание нового урока

1. Определи проект (см. `modules.json`, массив `projects`)
2. Определи номер — следующий после последнего элемента в `lessonIds` этого проекта
3. Создай папку: `p<N>-l<M>-<kebab-name>`
4. Добавь запись в `lessons.json` и ссылку в `lessonIds` проекта в `modules.json`
5. В шагах (`steps.json`) используй новый номер в видео-ссылке

## Требование: проверка изученного материала

## Требование: проверка изученного материала

Перед генерацией любых шагов (студенческих и менторских) агент ОБЯЗАН:

1. Прочитать `summary.md` **всех уроков модуля** — от первого до последнего.
2. Составить полную картину: какие концепции, операторы, типы, функции в каком уроке изучаются, а какие ещё не пройдены.
3. В шагах использовать **только то, что уже изучено** к моменту этого урока.
4. Если конструкция встречается в более позднем уроке — её НЕЛЬЗЯ использовать раньше.
5. Исключения из этого правила — только если пользователь явно разрешил.

Пример: в уроке 03 (математические операторы) ещё не пройдены `typeof` (урок 04), `NaN` (урок 04), приоритеты (урок 13), объекты (урок 15) — их нельзя использовать в шагах для урока 03.

## Требование: формат «Напиши программу, которая...»

В каждом уроке должен быть как минимум один шаг, где студент получает задачу на человеческом языке и пишет код самостоятельно. Это учит переводить требования в программу — ключевой навык Junior-разработчика.

Правила:
- Задача формулируется на русском (или казахском) языке, без готового кода
- Студент должен сам написать код от начала до конца
- Используются только уже изученные конструкции
- Для ввода данных до урока 06 используются переменные (let/const)
- Начиная с урока 06 можно использовать `prompt()`

## Разделение зон ответственности: студент vs ментор

### Студенческие шаги (самостоятельные задания)

**Цель:** легкое закрепление ключевой концепции урока.

Правила:
- Каждый шаг закрепляет **ровно одну концепцию** урока
- Формат: «предскажи вывод» для простых примеров ИЛИ «напиши программу по текстовой задаче» (одно действие)
- Студент должен справляться за 1–3 минуты
- Никаких неочевидных граничных случаев — это зона ментора
- Задача должна быть выполнима сразу после просмотра видео, без дополнительных пояснений

### Менторские файлы (для работы с ментором)

**Цель:** показать примеры, приближенные к реальной разработке, на ту же тему что в уроке, но сложнее студенческих.

Правила:
- Примеры должны быть **реалистичными** — такими, с которыми разработчик сталкивается в работе
- **Избегать** слишком пограничных / академических случаев: `NaN === NaN`, `null >= 0`, `new Boolean(false)`, `[] == false`, `parseInt('0x...')` и т.п.
- Примеры строятся по принципу: студенческие шаги = простые и закрепляющие, менторские = **углубление той же темы**, но с реальным контекстом
- **Можно дублировать тему студенческого задания**, но делать более сложную версию (больше условий, больше данных, реальный бизнес-сценарий)
- Показывают код, который может сломаться в реальном проекте, и объясняют, как писать правильно
- Ментор **не объясняет базу** — студент получил её на видео и в самостоятельных шагах; ментор показывает углублённые практические примеры

## Общая схема шагов в уроке

Каждый урок состоит из последовательности шагов, разделённых на две категории:

### 1. Шаги для студента (создаются через файл-предложение → утверждение → создание)

**Первый шаг — ссылка на внешний ресурс** (kind: `text`)
- Содержит ссылку на видео (Google Drive) или текстовый урок (learn-javascript)
- Короткое пояснение: о чём урок, что студент узнает
- **Не нужно** переписывать конспект из `lesson.md` в content
- `lesson.md` и `summary.md` нужны только **мне (агенту)** для понимания содержимого:
  - декомпозиция по проектам / урокам / шагам
  - генерация шагов для закрепления и шагов для ментора

**Последующие шаги — самостоятельный разбор** (kind: `text` или `code`)
Варианты:
- **Шаг-вопрос:** «Как ты думаешь, что будет, если выполнить в консоли следующий код: `...`? Ответь на вопрос и проверь правильность, выполнив его.»
- **Шаг-файл:** «Скачай файл и просмотри его содержимое. Ответь для себя, что будет происходить. Выполни и проверь.» (сейчас шаг-файл не реализован, используем `code` или `text`)
- **Шаг-тест:** вопрос с вариантами ответов (checkbox, radio) — пока не реализовано

Правила для самостоятельных шагов:
- Не выходят за пределы изученного урока
- Не рассматривают сложные и пограничные случаи (это к ментору)
- Закрепляют базовое понимание темы

### 2. Шаги-файлы для ментора (создаются сразу, без утверждения)

Хранятся в папке:
```
data/lessons/nur/<lesson-dir>/mentor-files/
```

Каждый файл — это полноценный шаг (kind: `file`), содержащий:
- **description** (поле `fileDescription`) — что это за файл, что с ним делать
- **Подсказки для ментора** — тоже в description (что обсудить, на что обратить внимание, как проводить занятие)
- **Код** (содержимое файла) — примеры с комментариями для **студентов**

**Структура файла:**
```js
// Пример 1: что будет результатом выполнения этой строки?
console.log('текст');

// Пример 2: обрати внимание — здесь неочевидный момент
let x = ...
```

Подсказки для ментора **не добавлять в код файла**. Они должны быть в `fileDescription` шага, который будет создан позже. В коде — только комментарии, адресованные студентам.

Соглашения:
- Файлы нумеруются: `01-<topic>.js`, `02-<topic>.js` и т.д.
- Примеры идут от простых к сложным
- Каждый пример может выводить:
    - промежуточные результаты в консоль
    - в конце — итоговый вывод / результат
- Комментарии в коде — вопросы к студентам: «что будет результатом выполнения этой строки?»
- Не части комментариями в коде, только для определенных случаев

### Размер менторских файлов

Менторский файл должен быть **20–30 строк** (примерно **10–15 `console.log`**) — этого достаточно для полноценного занятия. Если примеров больше — разбивай на несколько файлов.

**Почему именно так:**
- Занятие ментора длится ~15 минут на один файл — больше не успеют разобрать.
- 10–15 примеров покрывают тему без поверхностности.
- Студенты не теряют фокус.
- При необходимости можно сделать 2–3 файла по 15 строк вместо одного на 50.

**Не нужно:**
- Писать вопросы-подсказки в коде («что выведет?», «какой оператор выполнится первым?»). Вопросы задаёт ментор устно — они в `fileDescription`.
- Добавлять академические / пограничные случаи из серии `NaN === NaN`, `[] == false`.
- Устраивать «гонку примеров» — лучше меньше, но осмысленнее.

### Принцип «пасхалок»

Здраво вставляй в менторский файл 1–2 **пасхалки** — примеры, которые выглядят правильно, но падают (или работают неожиданно). Это заставляет студента думать, а не угадывать. Не нужно превращать в пасхалки весь файл — достаточно пары неочевидных моментов.

**Примеры пасхалок:**
- Обращение к переменной из другого блока — выглядит доступной, но `ReferenceError`
- Переменная с тем же именем во внешнем блоке — кажется что перезапишется, но нет
- `var` внутри блока — кажется что ограничен блоком, но «пробивает» его
- Переменная цикла после цикла — кажется что доступна, но нет (для `let`)
- Объект из функции без сохранения ссылки — кажется что жив, но GC удалит

Студент сначала разбирает файл самостоятельно (по мотивам урока), пытаясь предсказать вывод. Затем — совместный разбор с ментором. Пасхалки — те места, где предположения студентов чаще всего расходятся с реальностью.

### Стандартный формат занятия ментора

1. **До занятия:** студент получает файл, открывает, пытается предсказать поведение каждой строки
2. **На занятии:** ментор открывает файл в редакторе через проектор
3. Задаёт вопрос: «Что будет происходить, когда мы выполним этот файл?»
4. Студенты высказывают предположения
5. Ментор собирает мнения (можно таблицу ответов)
6. Выполняет код и разбирает результаты
7. Акцентирует внимание на неочевидных моментах — особенно там, где предположения студентов разошлись с реальностью

### Стандартное описание (fileDescription) для файла ментора

```
Скачай файл, открой в проекторе.

Подсказки для ментора:
1. Обсуди со студентами, что будет происходить при выполнении.
2. Обрати внимание студентов на участок <...>
3. Собери мнения, составь таблицу ответов (по студентам или по вариантам).
4. Выполни код, разбери результаты.
5. <дополнительные вопросы для обсуждения>
```

### Процесс создания шагов

1. Составить предложение шагов для студента (файл `proposed-steps.json`)
2. Пользователь утверждает / вносит правки
3. Создать шаги через `create-step` use case
4. Шаги-файлы для ментора создать сразу в `mentor-files/`
5. В будущем шаги ментора переедут в урок как полноценные шаги с kind: `file`

### Технические параметры create-step

```typescript
// kind: 'text'
{
  moduleId: string,      // UUID модуля
  lessonId: string,      // UUID урока
  description: string,   // описание шага (обязательно)
  kind: 'text',
  content?: string,      // содержимое (markdown)
}

// kind: 'code'
{
  moduleId: string,
  lessonId: string,
  description: string,
  kind: 'code',
  code: string,          // код для выполнения (обязательно)
  language?: string,     // язык (напр. 'javascript')
}

// kind: 'file'
{
  moduleId: string,
  lessonId: string,
  description: string,
  kind: 'file',
  fileName: string,       // имя файла
  fileMimeType: string,   // mime-тип
  fileSize: number,       // размер в байтах
  fileDescription: string, // описание + подсказки для ментора
}
```

## Правила составления шагов для практических (проектных) уроков

Практический урок — последний урок проекта, в котором студент применяет все изученные в проекте концепции для решения реальной задачи (например: «Расчёт зарплаты по отделу», «Рисуем ёлку в консоли», «reverseText через TDD»).

### Структура шагов практического урока

#### Шаг 1: Постановка задачи (kind: `text`)

Содержит полное описание задачи:
- **Что нужно сделать** — общее описание программы, которую предстоит написать
- **Исходные данные** — все входные данные (переменные, массивы, объекты, формулы)
- **Разбивка на этапы** — краткий обзор этапов (по одному на каждый последующий шаг)
- **Формулы и константы** — все необходимые для расчётов формулы

Пример хорошего первого шага:
```
### Что нужно сделать
Ты будешь писать программу для расчёта зарплаты всего отдела.
В отделе 4 сотрудника. Данные — массив и два объекта.
Задача разбита на 3 этапа.

### Исходные данные
[код с данными: массив employees, объекты salaries и workDays, константы]

### Формулы
1. Начислено = (оклад / WORK_DAYS_IN_MONTH) × дни
2. ОПВ = 10% от начисленного
...

### Этапы
| Этап | Что делаешь |
|------|-------------|
| 1 | Данные + проверка |
| 2 | Расчёт для каждого |
| 3 | Итоги по отделу |
```

#### Промежуточные шаги (kind: `text`)

По одному шагу на каждый этап. Каждый шаг содержит **только подсказки** — никакого готового кода.

**Жёсткое правило:** НИКОГДА не выдавать готовый код решения. Студент должен написать его сам.

Что можно давать:
- **Последовательность действий** на человеческом языке: «заведи переменную», «пройди циклом по массиву», «прибавь к сумматору»
- **Напоминание формул** (они уже даны в шаге 1, но можно кратко продублировать)
- **Подсказки о порядке операций**: «сначала посчитай начисленное, потом ОПВ, потом ИПН»
- **Имена переменных** которые нужно создать (но не их значения)

Что НЕЛЬЗЯ давать:
- ❌ Готовый код (ни одной строки)
- ❌ Точные выражения (`totalPayroll += salary`) — только словесное описание
- ❌ Реализацию условий и циклов — только указание что нужно условие/цикл

Пример хорошего промежуточного шага:
```
### Этап 2: расчёт для каждого сотрудника

Возьми код из этапа 1 и допиши внутрь цикла формулы расчёта зарплаты.

**Формулы:**
1. Начислено = (оклад / WORK_DAYS_IN_MONTH) × отработанные дни
2. ОПВ = начисленное × OPV_RATE
...

**Подсказки:**
- Все суммы округляй через Math.round()
- Оклад бери из salaries[сотрудник], дни — из workDays[сотрудник]
- Выведи результат для каждого сотрудника
```

#### Последний шаг: Проверка (kind: `text`)

Содержит тестовые данные для самопроверки:
- Ожидаемые результаты для каждого сотрудника
- Итоговые суммы по отделу
- Граничные случаи для проверки устойчивости

**Важно:** это только данные для сверки, а не автоматические тесты. Студент сам сравнивает вывод своей программы с ожидаемым.

### Принципы для практических уроков

1. **Студент пишет код сам** — от начала до конца. Шаги только направляют.
2. **Одна задача на урок** — не дробить на несвязанные упражнения.
3. **Постепенное усложнение** — каждый этап добавляет новый слой логики.
4. **Данные даны** — студент не придумывает входные данные, они уже есть в шаге 1.
5. **Проверка вручную** — тесты даны как ожидаемый вывод, без тестовых фреймворков.
