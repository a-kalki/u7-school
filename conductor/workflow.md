# Рабочий процесс проекта

## Руководящие принципы

1. **План — источник истины:** Вся работа должна отслеживаться в `plan.md`
2. **Технологический стек осознан:** Изменения в стеке должны быть задокументированы в `tech-stack.md` *до* реализации
3. **Разработка через тестирование (TDD):** Пиши модульные тесты перед реализацией функциональности
4. **Высокое покрытие кода:** Стремись к >80% покрытия для всех модулей
5. **Пользовательский опыт прежде всего:** Каждое решение должно ставить пользовательский опыт на первое место
6. **Non-interactive и CI-совместимость:** Предпочитай неинтерактивные команды. Используй `CI=true` для инструментов в режиме наблюдения (тесты, линтеры), чтобы гарантировать однократное выполнение.

## Рабочий процесс задач

Все задачи следуют строгому жизненному циклу:

### Стандартный рабочий процесс задачи

1. **Выбери задачу:** Возьми следующую доступную задачу из `plan.md` в последовательном порядке

2. **Отметь как «В процессе»:** Перед началом работы отредактируй `plan.md` и измени статус задачи с `[ ]` на `[~]`

3. **Напиши падающие тесты (Фаза Red):**
   - Создай новый тестовый файл для функции или исправления.
   - Напиши один или несколько модульных тестов, которые чётко определяют ожидаемое поведение и критерии приёмки для задачи.
   - Никогда не модифицируй существующие тесты без явного разрешения в задаче или запроси разрешение у пользователя.
   - **КРИТИЧНО:** Запусти тесты и убедись, что они падают, как и ожидалось. Это фаза «Red» TDD. Не продолжай, пока не получишь падающие тесты.

4. **Реализуй, чтобы тесты прошли (Фаза Green):**
   - Напиши минимальный объём кода, необходимый для прохождения падающих тестов.
   - Никогда не удаляй и не модифицируй код (реализацию) другого функционала без явного разрешения в задаче или запроси разрешение у пользователя.
   - Запусти тесты снова и убедись, что все тесты проходят. Это фаза «Green».

5. **Рефакторинг (Опционально, но рекомендуется):**
   - С защитой в виде проходящих тестов, отрефактори код реализации и тестовый код для улучшения читаемости, устранения дублирования и повышения производительности без изменения внешнего поведения.
   - Перезапусти тесты, чтобы убедиться, что они всё ещё проходят после рефакторинга.

6. **Проверь покрытие:** Запусти отчёты о покрытии с помощью инструментов проекта.
   ```bash
   bun test --coverage
   ```
   Цель: >80% покрытия для нового кода.

7. **Задокументируй отклонения:** Если реализация отличается от технологического стека:
   - **ОСТАНОВИ** реализацию
   - Обнови `tech-stack.md` с новым решением
   - Добавь заметку с датой, объясняющую изменение
   - Возобнови реализацию

8. **Закоммить изменения кода:**
   - Проиндексируй все изменения кода, связанные с задачей.
   - Предложи ясное, краткое сообщение коммита, например, `feat(ui): Создать базовую HTML-структуру для калькулятора`.
   - Выполни коммит.

9. **Прикрепи сводку задачи через Git Notes:**
   - **Шаг 9.1: Получи хеш коммита:** Получи хеш *только что сделанного коммита* (`git log -1 --format="%H"`).
   - **Шаг 9.2: Составь содержание заметки:** Создай подробную сводку выполненной задачи. Она должна включать имя задачи, сводку изменений, список всех созданных/изменённых файлов и основную причину изменений.
   - **Шаг 9.3: Прикрепи заметку:** Используй команду `git notes`, чтобы прикрепить сводку к коммиту.
     ```bash
     # Содержание заметки из предыдущего шага передаётся через флаг -m.
     git notes add -m "<содержание заметки>" <хеш_коммита>
     ```

10. **Получи и запиши SHA коммита задачи:**
    - **Шаг 10.1: Обнови план:** Прочитай `plan.md`, найди строку выполненной задачи, обнови её статус с `[~]` на `[x]` и добавь первые 7 символов хеша *только что сделанного коммита*.
    - **Шаг 10.2: Запиши план:** Запиши обновлённое содержимое обратно в `plan.md`.

11. **Закоммить обновление плана:**
    - **Действие:** Проиндексируй изменённый `plan.md`.
    - **Действие:** Закоммить это изменение с описательным сообщением (например, `conductor(plan): Отметить задачу 'Создать модель пользователя' как выполненную`).

### Протокол завершения фазы и создания контрольной точки

**Триггер:** Этот протокол выполняется сразу после завершения задачи, которая также закрывает фазу в `plan.md`.

1.  **Объяви о старте протокола:** Сообщи пользователю, что фаза завершена и протокол верификации и создания контрольной точки начался.

2.  **Обеспечь тестовое покрытие для изменений фазы:**
    -   **Шаг 2.1: Определи область фазы:** Чтобы определить файлы, изменённые в этой фазе, сначала найди начальную точку. Прочитай `plan.md`, чтобы найти SHA коммита Git *предыдущей* контрольной точки фазы. Если предыдущей контрольной точки нет, область — все изменения с первого коммита.
    -   **Шаг 2.2: Список изменённых файлов:** Выполни `git diff --name-only <sha_предыдущей_точки> HEAD`, чтобы получить точный список всех файлов, изменённых в этой фазе.
    -   **Шаг 2.3: Проверь и создай тесты:** Для каждого файла в списке:
        -   **КРИТИЧНО:** Сначала проверь его расширение. Исключи не-кодовые файлы (например, `.json`, `.md`, `.yaml`).
        -   Для каждого оставшегося кодового файла проверь, существует ли соответствующий тестовый файл.
        -   Если тестовый файл отсутствует, ты **должен** его создать. Перед написанием теста **сначала проанализируй другие тестовые файлы в репозитории**, чтобы определить правильное соглашение об именовании и стиль тестирования. Новые тесты **должны** проверять функциональность, описанную в задачах этой фазы (`plan.md`).

3.  **Запусти автоматические тесты с проактивной отладкой:**
    -   Перед выполнением ты **должен** объявить точную shell-команду, которую будешь использовать для запуска тестов.
    -   **Пример объявления:** «Я сейчас запущу автоматический набор тестов для проверки фазы. **Команда:** `CI=true bun test`»
    -   Выполни объявленную команду.
    -   Если тесты падают, ты **должен** сообщить пользователю и начать отладку. Ты можешь попытаться предложить исправление **не более двух раз**. Если тесты всё ещё падают после второго предложенного исправления, ты **должен остановиться**, сообщить о постоянной ошибке и запросить руководство пользователя.

4.  **Предложи подробный, действенный план ручной верификации:**
    -   **КРИТИЧНО:** Для создания плана сначала проанализируй `product.md`, `product-guidelines.md` и `plan.md`, чтобы определить пользовательские цели завершённой фазы.
    -   Ты **должен** создать пошаговый план, который проведёт пользователя через процесс верификации, включая все необходимые команды и конкретные ожидаемые результаты.
    -   План, который ты представляешь пользователю, **должен** следовать этому формату:

        **Для изменений фронтенда:**
        ```
        Автоматические тесты пройдены. Для ручной верификации, пожалуйста, выполни следующие шаги:

        **Шаги ручной верификации:**
        1.  **Запусти сервер разработки командой:** `bun run dev`
        2.  **Открой браузер по адресу:** `http://localhost:3000`
        3.  **Убедись, что видишь:** Страницу профиля пользователя с корректно отображаемыми именем и email.
        ```

        **Для изменений бэкенда:**
        ```
        Автоматические тесты пройдены. Для ручной верификации, пожалуйста, выполни следующие шаги:

        **Шаги ручной верификации:**
        1.  **Убедись, что сервер запущен.**
        2.  **Выполни следующую команду в терминале:** `curl -X POST http://localhost:8080/api/v1/users -d '{"name": "test"}'`
        3.  **Убедись, что получаешь:** JSON-ответ со статусом `201 Created`.
        ```

5.  **Дождись явной обратной связи от пользователя:**
    -   После представления подробного плана спроси пользователя: «**Соответствует ли это твоим ожиданиям? Пожалуйста, подтверди (да) или укажи, что нужно изменить.**»
    -   **ПРИОСТАНОВИСЬ** и жди ответа пользователя. Не продолжай без явного «да» или подтверждения.

6.  **Создай коммит контрольной точки:**
    -   Проиндексируй все изменения. Если на этом шаге изменений не было, сделай пустой коммит.
    -   Выполни коммит с ясным и кратким сообщением (например, `conductor(checkpoint): Контрольная точка конца Фазы X`).

7.  **Прикрепи аудируемый отчёт о верификации через Git Notes:**
    -   **Шаг 7.1: Составь содержание заметки:** Создай подробный отчёт о верификации, включая команду автоматического тестирования, шаги ручной верификации и подтверждение пользователя.
    -   **Шаг 7.2: Прикрепи заметку:** Используй команду `git notes` и полный хеш коммита из предыдущего шага, чтобы прикрепить полный отчёт к коммиту контрольной точки.

8.  **Получи и запиши SHA контрольной точки фазы:**
    -   **Шаг 8.1: Получи хеш коммита:** Получи хеш *только что созданного коммита контрольной точки* (`git log -1 --format="%H"`).
    -   **Шаг 8.2: Обнови план:** Прочитай `plan.md`, найди заголовок завершённой фазы и добавь первые 7 символов хеша коммита в формате `[checkpoint: <sha>]`.
    -   **Шаг 8.3: Запиши план:** Запиши обновлённое содержимое обратно в `plan.md`.

9. **Закоммить обновление плана:**
    - **Действие:** Проиндексируй изменённый `plan.md`.
    - **Действие:** Закоммить это изменение с описательным сообщением в формате `conductor(plan): Отметить фазу '<ИМЯ ФАЗЫ>' как завершённую`.

10.  **Объяви о завершении:** Сообщи пользователю, что фаза завершена и контрольная точка создана, с подробным отчётом о верификации, прикреплённым как git note.

### Контроль качества

Перед тем как отметить фазу как завершённую, проверь:

- [ ] Все тесты проходят
- [ ] Покрытие кода соответствует требованиям (>80%)
- [ ] Код следует руководствам по стилю проекта (как определено в `code_styleguides/`)
- [ ] Все публичные функции/методы документированы (например, docstrings, JSDoc, GoDoc)
- [ ] Типобезопасность обеспечена (например, type hints, типы TypeScript, типы Go)
- [ ] Нет ошибок линтинга или статического анализа (с использованием настроенных инструментов проекта)
- [ ] Корректно работает на мобильных устройствах (если применимо)
- [ ] Документация обновлена при необходимости
- [ ] Не внесено уязвимостей безопасности

## Команды разработки

### Настройка
```bash
# Установка зависимостей
bun install
```

### Ежедневная разработка
```bash
# Запуск всех тестов
bun test

# Запуск тестов конкретного пакета
bun test --filter @u7/course

# Форматирование кода
bunx biome format --write .

# Линтинг
bunx biome check --write .

# Проверка типов
bun run typecheck
```

### Перед коммитом
```bash
# Полная проверка: форматирование, линтинг, типы, тесты
bunx biome check --write . && bun run typecheck && bun test
```

## Требования к тестированию

### Модульное тестирование
- Каждый модуль должен иметь соответствующие тесты.
- Используй соответствующие механизмы настройки/очистки (например, fixtures, beforeEach/afterEach).
- Мокируй внешние зависимости.
- Тестируй как успешные, так и неуспешные сценарии.

### Интеграционное тестирование
- Тестируй полные пользовательские сценарии
- Проверяй транзакции базы данных
- Тестируй аутентификацию и авторизацию
- Проверяй отправку форм

### Мобильное тестирование
- Тестируй на реальном iPhone, когда возможно
- Используй инструменты разработчика Safari
- Тестируй сенсорные взаимодействия
- Проверяй адаптивные макеты
- Проверяй производительность на 3G/4G

## Процесс код-ревью

### Чеклист самопроверки
Перед запросом ревью:

1. **Функциональность**
   - Функция работает как указано в спецификации
   - Граничные случаи обработаны
   - Сообщения об ошибках понятны пользователю

2. **Качество кода**
   - Соответствует руководству по стилю
   - Применён принцип DRY
   - Понятные имена переменных/функций
   - Уместные комментарии

3. **Тестирование**
   - Модульные тесты всеобъемлющи
   - Интеграционные тесты проходят
   - Покрытие адекватное (>80%)

4. **Безопасность**
   - Нет захардкоженных секретов
   - Присутствует валидация ввода
   - Предотвращены SQL-инъекции
   - На месте защита от XSS

5. **Производительность**
   - Запросы к базе данных оптимизированы
   - Изображения оптимизированы
   - Кеширование реализовано где нужно

6. **Мобильный опыт**
   - Сенсорные цели адекватны (44x44px)
   - Текст читаем без масштабирования
   - Производительность приемлема на мобильных
   - Взаимодействия ощущаются нативными

## Руководство по коммитам

### Формат сообщения
```
<тип>(<область>): <описание>

[опциональное тело]

[опциональный footer]
```

### Типы
- `feat`: Новая функциональность
- `fix`: Исправление бага
- `docs`: Только документация
- `style`: Форматирование, пропущенные точки с запятой и т.д.
- `refactor`: Изменение кода, которое не исправляет баг и не добавляет функциональность
- `test`: Добавление недостающих тестов
- `chore`: Задачи обслуживания

### Примеры
```bash
git commit -m "feat(auth): Добавить функциональность 'запомнить меня'"
git commit -m "fix(posts): Исправить генерацию excerpt для коротких постов"
git commit -m "test(comments): Добавить тесты для лимитов emoji-реакций"
git commit -m "style(mobile): Улучшить сенсорные цели кнопок"
```

## Определение готовности

Задача считается завершённой, когда:

1. Весь код реализован согласно спецификации
2. Модульные тесты написаны и проходят
3. Покрытие кода соответствует требованиям проекта
4. Документация завершена (если применимо)
5. Код проходит все настроенные проверки линтинга и статического анализа
6. Красиво работает на мобильных (если применимо)
7. Заметки о реализации добавлены в `plan.md`
8. Изменения закоммичены с правильным сообщением
9. Git note со сводкой задачи прикреплён к коммиту

## Чрезвычайные процедуры

### Критический баг в продакшене
1. Создай ветку hotfix от main
2. Напиши падающий тест для бага
3. Реализуй минимальное исправление
4. Тщательно протестируй, включая мобильные
5. Разверни немедленно
6. Задокументируй в plan.md

### Потеря данных
1. Останови все операции записи
2. Восстанови из последнего бэкапа
3. Проверь целостность данных
4. Задокументируй инцидент
5. Обнови процедуры бэкапа

### Нарушение безопасности
1. Немедленно смени все секреты
2. Проверь логи доступа
3. Исправь уязвимость
4. Уведоми затронутых пользователей (если есть)
5. Задокументируй и обнови процедуры безопасности
