# TDD и функция len()

**Краткое содержание:**
Урок посвящён практике TDD (Test-Driven Development) — разработке через тестирование. Студент проходит полный цикл red → green → refactor на примере функции `len(str)`, которая считает длину строки без использования `.length`. Урок пошагово показывает весь воркфлоу: от создания файлов до первого коммита.

### Подробный конспект

#### 1. Что такое TDD

**TDD (Test-Driven Development)** — подход к разработке, при котором ты **сначала пишешь тест, а потом код**. Цикл состоит из трёх шагов:

1. **RED (красный)** — написать тест, который падает (функции ещё нет или она не реализована)
2. **GREEN (зелёный)** — написать минимальный код, чтобы тест прошёл
3. **REFACTOR (синий)** — улучшить код, не ломая тесты

**Почему TDD:**
- Ты точно знаешь, что код делает то, что нужно
- Ты не пишешь лишнего кода — только то, что покрыто тестами
- При доработках тесты сразу покажут, если ты что-то сломал
- Каждый тест — это документация: что функция принимает и что возвращает

#### 2. Инструмент: bun:test

Мы используем встроенный тестовый фреймворк Bun — `bun:test`. Он не требует установки (идёт вместе с Bun).

**Основные функции:**
- `describe(название, () => { ... })` — группирует тесты
- `test(название, () => { ... })` — отдельный тест
- `expect(значение).toBe(ожидание)` — проверка равенства
- `expect(() => fn()).toThrow(ТипОшибки)` — проверка, что функция выбрасывает ошибку

#### 3. Пошаговый воркфлоу TDD для len(str)

##### Шаг 1: Создай файл с функцией-заглушкой

Создай в проекте `js-algorithms` файл `len.js`:

```javascript
// len.js
// ЗАГЛУШКА — пока возвращает пустую строку вместо числа.
// На шаге RED тест упадёт, потому что ожидается число, а не строка.

export function len(str) {
  return '';
}
```

**Почему возвращаем пустую строку?** Потому что это «неправильный» тип — тест ожидает число. Если бы мы вернули `0`, тест `len('hello') === 5` всё равно упал бы (5 ≠ 0), но идея TDD в том, что заглушка должна быть максимально простой и заведомо неправильной.

##### Шаг 2: Создай файл с тестами

Создай файл `len.test.js`:

```javascript
// len.test.js
// Импортируем всё необходимое для тестов

// describe и test — из bun:test (встроенный тестовый фреймворк Bun)
import { describe, test, expect } from 'bun:test';

// Импортируем тестируемую функцию (путь ./len.js — тот же каталог)
import { len } from './len.js';

// Подсказка: импорты можно вставить через автокомплит редактора!
// Начни печатать "import { describe" — редактор предложит варианты.
// Для импорта своего файла начни с "./" и редактор покажет файлы в папке.

describe('Тесты len', () => {
  test('должна вернуть 5 для строки "hello"', () => {
    expect(len('hello')).toBe(5);
  });
});
```

##### Шаг 3: Запусти тесты — RED

```bash
bun test
```

**Ожидаемый результат — тест падает (красный):**
```
✗ должна вернуть 5 для строки "hello" [fail]
expect(received).toBe(expected)

Expected: 5
Received: ""
```

Это нормально! Мы на шаге RED — тест написан, код ещё не реализован. Падающий тест подтверждает, что тест рабочий и ловит несоответствие.

##### Шаг 4: Напиши минимальную реализацию — GREEN

Меняем `len.js` — заменяем заглушку на реальный код:

```javascript
// len.js
// Реализация: считаем символы циклом до первого undefined

export function len(str) {
  // Проверка типа (изучили в уроке 1.3)
  if (typeof str !== 'string') {
    throw new TypeError('Аргумент str должен быть строкой');
  }

  let count = 0;
  for (let i = 0; str[i] !== undefined; i++) {
    count++;
  }
  return count;
}
```

Запускаем `bun test`:

```
✓ должна вернуть 5 для строки "hello" [pass]
```

**GREEN!** Тест проходит — реализация работает.

##### Шаг 5: Добавляй тесты по одному — каждый новый случай = новый тест

Теперь, когда базовый случай работает, будем добавлять тесты на крайние случаи **по одному**. Это важная часть TDD: каждый новый тест — это новая «спецификация» того, что должна делать функция.

**Порядок добавления:** напиши тест → убедись что он упал (RED) → допиши код → убедись что прошёл (GREEN) → коммит.

###### Шаг 5a — Пустая строка

Добавь тест: **«должна вернуть 0 для пустой строки»**.

Реализация уже справляется (цикл не выполнится ни разу, count останется 0). Тест должен сразу стать зелёным.

###### Шаг 5b — Строка из пробелов

Добавь тест: **«должна вернуть 3 для строки из трёх пробелов»**.

Пробелы — такие же символы, как и буквы. Реализация уже считает их. Тест должен сразу стать зелёным.

###### Шаг 5c — Кириллица

Добавь тест: **«должна корректно считать кириллицу»** (например, `len('привет')` → 6).

Кириллические символы ничем не отличаются от латинских при подсчёте. Тест должен сразу стать зелёным.

###### Шаг 5d — Проверка типа: число

Добавь тест: **«должна выбросить TypeError если аргумент — число»** (например, `len(123)`).

Используй `expect(() => len(123)).toThrow(TypeError)`. Реализация уже содержит проверку `typeof`, тест должен быть зелёным.

###### Шаг 5e — Проверка типа: null и undefined

Добавь тесты: **«должна выбросить TypeError если аргумент — null»** и **«должна выбросить TypeError если аргумент — undefined»**.

Убедись, что оба случая обрабатываются. Тесты должны быть зелёными.

После добавления всех тестов запусти `bun test` — все 6 тестов должны гореть зелёным. Refactor не требуется — реализация уже покрывает все случаи.

##### Шаг 6: Первый коммит

Когда все тесты проходят — время сохранить изменения в Git:

```bash
git add len.js len.test.js
git commit -m "Реализовать функцию len(str) через TDD"
```

**Правила хорошего commit-сообщения:**
- На русском или английском — как принято в команде
- Глагол в повелительном наклонении: «Добавить», «Исправить», «Реализовать»
- Кратко (до 72 символов в первой строке)
- Понятно без контекста: что сделано, а не как
