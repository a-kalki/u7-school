# Шаги урока: TDD и функция len()

---

### TDD

**kind:** `text`

**TDD (Test-Driven Development)** — подход к разработке, при котором ты **сначала пишешь тест, а потом код**. Цикл состоит из трёх шагов:

1. **RED** — написать тест, который падает (функции ещё нет или она не реализована)
2. **GREEN** — написать минимальный код, чтобы тест прошёл
3. **REFACTOR** — улучшить код, не ломая тесты

**Почему TDD:**
- Ты точно знаешь, что код делает то, что нужно
- Ты не пишешь лишнего кода — только то, что покрыто тестами
- При доработках тесты сразу покажут, если ты что-то сломал

Мы используем встроенный тестовый фреймворк Bun — `bun:test`:
- `describe(название, () => { ... })` — группирует тесты
- `test(название, () => { ... })` — отдельный тест
- `expect(значение).toBe(ожидание)` — проверка равенства
- `expect(() => fn()).toThrow(ТипОшибки)` — проверка, что функция выбрасывает ошибку

### TDD: создай заглушку

**kind:** `text`

Создай в проекте `js-algorithms` файл `len.js`. Функция пока будет **заглушкой** — заведомо неправильной. На текущей стадии функция должна вернуть undefined.

Тест будет ожидать **число**, поэтому гарантированно упадёт. В TDD заглушка должна быть максимально простой и падать на первом же тесте.

---

### TDD: создай тест

**kind:** `text`

Создай файл `len.test.js`:

```javascript
// len.test.js
import { describe, test, expect } from 'bun:test';
import { len } from './len.js';

describe('Тесты len', () => {
  test('должна вернуть 5 для строки "hello"', () => {
    // тут твоя проверка работы функции
  });
});
```

**Подсказка по импортам:**
1. Вариант 1: начни печатать `import { describe` — редактор предложит автокомплит из `bun:test`;
1. Вариант 2: не надо явно делать импорт, просто начни писать вызов функции `descr...`, автокомплит предложит тебе варианты, выбери `describe` и нажми "Enter". Редактор сам добавит в начале файла `import ...`.

### TDD: RED

**kind:** `text`

Запусти тесты:

```
bun test
```

Ожидаемый вывод — тест падает:

```
✗ должна вернуть 5 для строки "hello" [fail]
Expected: 5
Received: undefined
```

Это нормально! Мы на шаге RED — тест написан, код ещё не реализован. Падающий тест **подтверждает**, что тест рабочий и ловит несоответствие.

Запусти `bun test` — тест должен стать зелёным ✅.

### TDD: добавь тесты на граничные случаи

**kind:** `text`

Добавь в `len.test.js` тесты для следующих случаев. Каждый случай — отдельный `test(...)`.

Тестовые случаи:
- Должна вернуть `0` для пустой строки (`''`)
- Должна вернуть `3` для строки из трёх пробелов (`'   '`)
- Должна корректно считать кириллицу — `len('привет')` → `6`

После добавления каждого теста запускай `bun test`. Реализация уже должна справляться со всеми тремя случаями — тесты должны быть зелёными. Если нет, исправляй реализацию.

---

### TDD: тесты на исключения

**kind:** `text`

Добавь тесты, проверяющие что функция **выбрасывает ошибку** на некорректных входных данных.

**Вспомни воркфлоу TDD для каждого теста:**
1. Добавь тест — `bun test` → тест **должен упасть** (RED), потому что в реализации ещё нет такой проверки
2. Допиши проверку в `len.js` → тест **должен пройти** (GREEN)
3. Убедись что предыдущие тесты не сломались

Тестовые случаи:
- Должна выбросить `TypeError` если аргумент — число (`123`)
- Должна выбросить `TypeError` если аргумент — `null`, `undefined`

Для проверки исключений используй:
```javascript
expect(() => len(123)).toThrow(TypeError);
// или если хочешь проверить конкретный случай
expect(() => len(123)).toThrow('Аргумент должен быть строкой');
```

Запусти `bun test` — все тесты должны гореть зелёным ✅.

### TDD: первый коммит

**kind:** `text`

Когда все тесты проходят — сохрани изменения в Git.

**Добавление файлов в stage**

Есть несколько вариантов команды `git add`:

```
# Добавить конкретные файлы (рекомендуется — точно знаешь что коммитишь)
git add len.js len.test.js

# Добавить все файлы в текущей папке и подпапках
git add .

# Добавить все файлы в репозитории (где бы ты ни находился)
git add -A
```

Перед коммитом всегда проверяй что попало в stage:

```
git status
```

Зелёным подсвечены файлы готовые к коммиту, красным — изменённые но не добавленные.

**Создание коммита**

```
git commit -m "Реализована функция len(str) через TDD"
```

Флаг `-m` позволяет написать сообщение прямо в командной строке. Если опустить `-m`, Git откроет текстовый редактор для ввода сообщения.

Правила хорошего сообщения:
- Глагол в прошедшем времени или инфинитиве: «Реализована», «Добавлена», «Исправлена»
- Кратко — первая строка до 72 символов
- Понятно без контекста: **что** сделано, а не **как**

Примеры:
```
✅ Реализована функция len(str) через TDD
✅ Добавлены тесты на граничные случаи для len
❌ fix
❌ изменения
```

Проверь результат:

```
git log --oneline
```
