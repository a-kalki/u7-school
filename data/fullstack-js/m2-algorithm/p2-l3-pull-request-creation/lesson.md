# Pull Request: создание и описание

**Краткое содержание:**
Студент изучает, что такое Pull Request (PR), зачем нужно осмысленное описание, и создаёт свой первый PR из кода проекта 1.

### 1. Что такое Pull Request

**Pull Request (PR)** — это предложение внести изменения в код. Ты говоришь: «Я написал код, посмотрите его и добавьте в основную ветку».

В крупных компаниях (Google, Microsoft, Yandex) **ни одна строка кода не попадает в основную ветку без PR и ревью**. Это стандарт индустрии.

**Как это работает:**
1. Ты сделал ветку, написал код, запушил на GitHub
2. Создаёшь PR: выбираешь ветку-источник (твою) и ветку-назначение (обычно `main`)
3. Назначаешь ревьюеров — коллег, которые проверят твой код
4. Ревьюеры оставляют комментарии: что нужно поправить, что улучшить
5. Ты вносишь правки, пушишь обновления (PR обновляется автоматически)
6. Когда ревьюеры одобрят — ты (или они) нажимаешь Merge
7. Твой код попадает в основную ветку

**Почему PR — это не просто «залить код»:**
- Код-ревью находит ошибки, которые ты не заметил
- Коллеги предлагают лучшие решения (и ты учишься)
- Обсуждение в PR остаётся в истории — через полгода можно понять, почему сделали именно так
- Равномерно распределяется знание о коде (а не «это Вася писал, только он и знает как оно работает»)

### 2. GitHub Flow

**GitHub Flow** — упрощённый рабочий процесс, принятый на GitHub:

1. **Создать ветку** от `main`
2. **Добавить коммиты**
3. **Открыть Pull Request**
4. **Обсудить и доработать** код по результатам ревью
5. **Слить (merge)** ветку в `main`
6. **Удалить ветку**

В этом модуле мы используем именно GitHub Flow.

### 3. Создание PR на GitHub

Поскольку проект 1 ты делал в единственной ветке `main` (не создавая feature-ветку), PR создаётся из `main` в `main`. Это допустимо для первого PR, но в следующих проектах мы будем использовать feature-ветки.

**Шаги:**
1. Перейди на страницу своего репозитория `js-algorithms` на GitHub
2. Нажми на вкладку **Pull requests**
3. Нажми зелёную кнопку **New pull request**
4. Выбери ветки:
   - **base:** `main` (куда вливаем)
   - **compare:** `main` (откуда берём)
5. GitHub покажет, что ветки одинаковые, но PR всё равно можно создать. Нажми **Create pull request**

### 4. Осмысленное описание PR

Хорошее описание PR экономит время ревьюеров. Они сразу понимают, что ты сделал и на что обратить внимание.

**Шаблон описания:**

```markdown
## Что сделано

Реализованы 7 функций посимвольной обработки строк:
- `len(str)` — длина строки
- `isEqual(a, b)` — посимвольное равенство
- `isNotEqual(a, b)` — отрицание равенства
- `isMore(a, b)` — лексикографическое «больше»
- `isLess(a, b)` — лексикографическое «меньше»
- `isMoreOrEqual(a, b)` — больше или равно
- `isLessOrEqual(a, b)` — меньше или равно

## Как запустить тесты

```
bun test
```

Все 7 функций покрыты тестами.

## На что обратить внимание

- Обработка граничных случаев: пустые строки, кириллица, спецсимволы
- Проверка типов аргументов (throw TypeError)
- Лексикографическое сравнение реализовано по кодам символов (ASCII/Unicode)
```

**Принципы хорошего описания:**
- **Что сделано** — краткий список изменений (лучше чеклистом)
- **Как запустить** — команда для запуска/тестирования
- **На что обратить внимание** — что ревьюерам нужно проверить особенно тщательно
- Не нужно пересказывать каждую строчку кода — ревьюеры увидят diff

### 5. Заголовок PR

Заголовок — первое, что видят ревьюеры. Хороший заголовок:
- Отражает суть изменений (как commit-сообщение)
- На русском или английском — как в команде

**Примеры заголовков:**
```
✅ Реализовать 7 функций сравнения строк с TDD
✅ Проект 1: посимвольное сравнение строк и Git
❌ мой код
❌ Сделал всё
```
