### Создание первого PR

**kind:** `text`

Создай свой первый Pull Request на GitHub.

**На GitHub:**
1. Перейди в репозиторий `js-algorithms`
2. Вкладка **Pull requests** → **New pull request**
3. Выбери `base: main` ← `compare: main` (да, обе `main` — для первого PR это нормально)
4. Нажми **Create pull request**

**Заголовок:**
```
Проект 1: 7 функций сравнения строк с TDD
```

**Описание — заполни по шаблону:**

```
## Что сделано

Реализованы 7 функций посимвольной обработки строк:
- [ ] len(str) — длина строки
- [ ] isEqual(a, b) — посимвольное равенство
- [ ] isNotEqual(a, b) — отрицание равенства
- [ ] isMore(a, b) — лексикографическое «больше»
- [ ] isLess(a, b) — лексикографическое «меньше»
- [ ] isMoreOrEqual(a, b) — больше или равно
- [ ] isLessOrEqual(a, b) — меньше или равно

## Как запустить тесты

bun test

## На что обратить внимание

- Обработка граничных случаев: пустые строки, кириллица
- Проверка типов аргументов (throw TypeError)
- Лексикографическое сравнение по кодам символов
```

### Просмотр PR глазами ревьюера

**kind:** `text`

Посмотри на созданный PR глазами ревьюера:

1. Перейди на вкладку **Files changed** — это diff: красным показаны удалённые строки, зелёным — добавленные
2. Пролистай изменения: видно ли по diff, что именно изменилось?
3. Представь, что ты видишь этот PR впервые. Понятно ли из описания, что сделано и как это проверить?

Если что-то непонятно — дополни описание PR (кнопка **Edit** рядом с заголовком).

### Вопросы на понимание

**kind:** `text`

Ответь на вопросы:

1. Ты создал PR из `main` в `main`. Это работает, но неудобно для ревью — почему? (Подсказка: представь, что в `main` уже есть изменения от другого разработчика)
2. Представь, что ревьюер попросил исправить функцию `isMore`. Ты внёс правки локально, сделал коммит и `git push`. Нужно ли создавать новый PR? Почему?
3. Посмотри на вкладку **Files changed**. Почему GitHub показывает diff, а не просто список файлов? Зачем ревьюеру видеть, какие именно строки изменились?
