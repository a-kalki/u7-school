# Шаги урока: JSDoc — документирование функций

---

### Две цели урока: JSDoc и работа без PR

**kind:** `text`

У этого урока две цели:

1. **Научиться документировать код через JSDoc** — стандарт, принятый во всей JS-экосистеме
2. **Научиться делать некритичные изменения напрямую в `main`** — без Pull Request и код-ревью, когда это уместно

Код читают намного чаще, чем пишут. **JSDoc** — это комментарии `/** ... */` перед функцией, которые описывают: что функция делает, что принимает, что возвращает, какие ошибки выбрасывает.

**Изучи материал:**

- [JSDoc](https://learn.javascript.ru/jsdoc) — русскоязычное введение, прочитай всю статью
- [Официальная документация JSDoc](https://jsdoc.app/) — все поддерживаемые теги (для справки, пока можно просто глянуть)
[Markdown за 5 минут](https://learn.javascript.ru/markdown) — Markdown — это облегченный язык разметки текста, созданный для быстрого и удобного форматирования документов с помощью обычных символов

**Ключевые теги:**

| Тег | Назначение |
|-----|-----------|
| `@param {type} name — описание` | Параметр функции |
| `@returns {type} — описание` | Возвращаемое значение |
| `@throws {type} — описание` | Исключение |
| `@example` | Пример использования |

### Хороший и плохой JSDoc: примеры

**kind:** `text`

Сначала посмотри, как выглядит хороший JSDoc, а как — плохой:

**Плохо (так не пишем):**
```javascript
/** @param a @param b @returns boolean */
function isEqual(a, b) { ... }
```

**Хорошо (так пишем):**
```javascript
/**
 * Посимвольное сравнение двух строк на равенство. (Плохо, говорит как делает)
 * Возвращает равны ли строки a и b. (Это получше, говорит что делает)
 *
 * @param {string} a — первая строка
 * @param {string} b — вторая строка
 * @returns {boolean} — true если строки равны, false если нет
 * @throws {TypeError} — если любой из аргументов не строка
 *
 * @example
 *   isEqual('hello', 'hello'); // true
 *   isEqual('hi', 'hello');    // false
 */
export function isEqual(a, b) { ... }
```

Обрати внимание на:
- Две звёздочки `/**` в начале
- Пустые строки между группами тегов (читаемость)
- Тире `—` после имени параметра и типа возврата
- Отступ в 2 пробела для примеров в `@example`

### Документируем функции и пушим напрямую в main

**kind:** `text`

Теперь **применим знания на практике** — добавим JSDoc ко всем написанным функциям и отправим изменения на GitHub напрямую в `main` без Pull Request и процедуры code-review.

> Данным уроком подразумевается, что ты уже замерджил первый проект в `main`, т.е. завершил полностью code-review по первому PR и слил его в основную ветку. Если нет, то прочитай шаг и посоветуйся с ментором: как быть дальше.

Сначала убедись, что текущая директория чистая:
```
git status
```

Если `git status` показывает незакоммиченные изменения — закоммить их.

Перейди в `main`.

```
git switch main
```

Открой редактор и посмотри есть ли файлы из проекта 1 (len(), isEqual(), ...).

Если нет, значит ветка `main` у тебя отстал от `origin:main`, нужно их подтянуть из удаленного репозитория (gitHub-а). Чтобы убедиться в этом набери `git status`, сообщение должно сказать, что в удаленном репозитории есть новые коммиты.

```
git pull
```

Убедись в редакторе, что файлы появились.

Добавь докстроки для двух «разминочных» функций `len(), isEqual()`, сверяясь с примером выше.

После этого открой `len.test.ts` и наведи мышку на вызов функции — VS Code (твой редактор IDE) должен показать описание из JSDoc.

Теперь задокументируй остальные функции залитые в `main`.

Когда всё готово — закоммить и отправь на GitHub:

```
git add -A
git commit -m "Добавлен JSDoc для всех функций первого проекта"
git push
```

Перейди в свой репозиторий на GitHub и открой пару файлов. Убедись, что JSDoc-комментарии видны прямо в коде на GitHub.

**Стоп. Оглянись назад.** Ты только что:
- Переключился в `main` (не создавал feature-ветку)
- Внёс изменения во множество файлов
- Закоммитил и запушил **напрямую в `main`** — без Pull Request, без ревью, без аппрувов

Допустимо ли это?

Все зависит от:
- работаешь ли ты один?;
- что было изменено, насколько "рискованные" эти изменения, могут ли они сломать приложение;
- разрешено ли это в той команде где ты работаешь.

Сейчас это твой личный репозиторий и через код ревью делай то, что требуется по заданиям. Остальные изменения на твое усмотрение, если хочешь можешь просить у сокурсников дополнительные код-ревью.

### Правило: JSDoc сразу после заглушки в TDD-цикле

**kind:** `text`

Запомни правило: начиная с этого проекта, **каждая** новая функция должна быть задокументирована в JSDoc **сразу после заглушки**, до начала реализации — это часть TDD-цикла.

Почему именно так? Потому что JSDoc заставляет тебя **сначала подумать о контракте функции**: что она принимает, что возвращает, какие ошибки возможны. Это улучшает дизайн ещё до того, как ты написал первую строку реализации.

### Вопросы на понимание

**kind:** `text`

Ответь на вопросы. Если не знаешь ответ — подумай, проверь догадки на своих же файлах.

**1.** Открой любой `.test.js` файл и наведи мышку на вызов функции, для которой ты написал JSDoc. Что показывает VS Code? Откуда он это взял?

**2.** Представь: ты вернулся к своему коду через месяц. Ты открываешь `is-more.js` и видишь только:
```javascript
export function isMore(a, b) {
  if (typeof a !== 'string') throw new TypeError('...');
  // ... 20 строк алгоритма
}
```
Без JSDoc. Сколько секунд ты потратишь, чтобы понять: что принимает функция, что возвращает (`true`/`false` или `1`/`-1`?), какие ошибки кидает? А если JSDoc есть — ты читаешь 5 строк комментария и уже знаешь ответ.

**3.** Ты отправляешь свой код ревьюеру. Он открывает PR и видит 8 новых функций. Без JSDoc ревьюеру придётся читать тело каждой функции, чтобы понять что она делает. С JSDoc — пробегает глазами по описанию и проверяет только логику. Как ты думаешь, в каком случае ревьюер:
- Быстрее даст аппрув?
- Найдёт больше ошибок в логике (а не в понимании)?
- Будет меньше раздражён?

**4.** JSDoc — это не только для людей. Инструменты вроде `jsdoc-to-markdown` могут сгенерировать из твоих комментариев полноценный сайт с документацией. Попробуй мысленно представить: у тебя 51 функция в проекте. Как быстрее объяснить новому разработчику что делает каждая — дать ему читать 51 файл с кодом или открыть сгенерированную страницу с сигнатурами и описаниями?
