# Ветка для исправлений по ревью

**Краткое содержание:**
Урок закрепляет навыки работы с замечаниями ревью, полученные в проекте 2. Теперь ты применяешь их к реальному PR с функциями поиска подстрок: создаёшь fix-ветку, вносишь правки, пушишь и запрашиваешь повторное ревью.

> Процесс ревью и правок ты уже проходил в уроках 2.4–2.5. Здесь — практическое применение тех же механик. Если забыл детали — вернись к тем урокам и освежи в памяти.
>
> ⏳ **Шаги этого урока выполняются не сразу**, а когда ревьюеры оставят комментарии к PR проекта 3. Прочитай конспект сейчас, жми «Выполнено» в шагах и возвращайся когда придёт ревью.

### 1. Почему правки — в отдельной ветке

Когда ревьюер оставляет замечания, не стоит править код прямо в feature-ветке. Причины:

- Feature-ветка может понадобиться в исходном виде (сравнить «до» и «после»)
- Если правок много — проще отслеживать их в отдельной ветке
- Это хорошая практика: каждая задача — в своей ветке

**Рабочий процесс:**
1. Получаешь комментарии ревью
2. Создаёшь fix-ветку **от feature-ветки**: `git checkout -b fix/review-comments feat/substring-search`
3. Вносишь правки
4. Пушишь fix-ветку
5. Отвечаешь на комментарии (отмечаешь resolved)
6. Запрашиваешь повторное ревью

### 2. Типичные замечания ревью

Что могут попросить исправить ревьюеры (примеры из проекта 3):

- **Имена:** функция `reverse` понятна, а вот аргументы — может, `str` и `search` можно назвать понятнее?
- **Проверки типов:** везде ли есть `throw new TypeError`?
- **Граничные случаи:** что будет, если в `repeat` передать `count = Infinity`?
- **Дублирование:** в `startsWith` и `endsWith` похожий код проверки типов — можно вынести в отдельную функцию?
- **Комментарии/JSDoc:** функции не задокументированы (JSDoc будет в проекте 4, но ревьюер может спросить «а где описание?»)

### 3. Как отвечать на комментарии

**Правила хорошего тона:**
- На каждый комментарий — ответ. Даже если просто «Исправил» или «Done»
- Если не согласен — объясни почему, со ссылкой на документацию или логику
- Не принимай критику на свой счёт — ревьюер смотрит на код, а не на тебя
- Благодари за найденные проблемы — это экономит время later

**Пример ответа:**
> Поправил, вынес проверку типов в хелпер `ensureString`. Спасибо!

**Пример несогласия:**
> Оставил `str` и `search`, потому что это стандартные имена аргументов в документации JS. Менять на `haystack` и `needle` — менее привычно.

### 4. Технический процесс

**Создание fix-ветки:**
```bash
git checkout feat/substring-search     # перейти в feature-ветку
git checkout -b fix/review-comments    # создать fix-ветку от неё
```

**Внесение правок:**
```bash
# ... редактируешь файлы ...
git add .
git commit -m "Правки по ревью: проверка на Infinity в repeat"
```

**Пуш fix-ветки:**
```bash
git push -u origin fix/review-comments
```

**Обновление PR:**
- Если ты создал fix-ветку от feature и запушил — нужно создать новый PR из `fix/review-comments` в `feat/substring-search` (это PR в PR!)
- Альтернатива (проще): править прямо в `feat/substring-search`, коммитить и пушить — PR обновится автоматически
- Второй подход проще для первого опыта, используй его

**Ответ на комментарии и resolved:**
- Под каждым комментарием ревьюера напиши ответ
- Если правка внесена — нажми **Resolve conversation**

**Запрос повторного ревью:**
- В правой панели PR нажми иконку «смена» (re-request review) рядом с именем ревьюера
