### Практика: fast-forward merge

**kind:** `text`

> **Windows:** все команды ниже выполняй в **Git Bash**. Обычный cmd или PowerShell не подойдут.

Проверь fast-forward слияние на практике. Эксперименты будем проводить в **отдельной** ветке, чтобы не загрязнять рабочую `feat/substring-search`.

**1. Создай экспериментальную ветку:**
```
git switch -c experiment/merge-lab
```

**2. Создай тестовый файл и закоммить его:**
```
echo "console.log('hello from feature');" > test-merge.js
git add test-merge.js
git commit -m "Добавлен test-merge.js в experiment/merge-lab"
```

**3. Переключись на `main`:**
```
git switch main
```

**4. Выполни слияние:**
```
git merge experiment/merge-lab
```

Ты должен увидеть `Fast-forward` — потому что в `main` не было новых коммитов.

**5. Проверь историю:**
```
git log --oneline
```
Коммит из `experiment/merge-lab` теперь в `main`.

### Практика: создай конфликт и разреши его

**kind:** `text**

Смоделируем ситуацию, которая приводит к конфликту.

**1. Находясь в `main`, создай файл `greeting.js`:**
```
echo "console.log('Версия из main');" > greeting.js
git add greeting.js
git commit -m "greeting.js из main"
```

**2. Создай новую ветку и измени тот же файл:**
```
git switch -c feat/greeting-update
echo "console.log('Версия из feat/greeting-update');" > greeting.js
git add greeting.js
git commit -m "greeting.js изменён в feat/greeting-update"
```

**3. Вернись в `main` и измени тот же файл по-другому:**
```
git switch main
echo "console.log('Другая версия из main');" > greeting.js
git add greeting.js
git commit -m "greeting.js изменён в main"
```

**4. Попробуй слить `feat/greeting-update` в `main`:**
```
git merge feat/greeting-update
```
Git выдаст `CONFLICT (content): Merge conflict in greeting.js` и `Automatic merge failed`.

**5. Посмотри содержимое `greeting.js`:**
```
cat greeting.js
```
Ты увидишь маркеры конфликта `<<<<<<<`, `=======`, `>>>>>>>`.

**6. Разреши конфликт:** открой файл в редакторе, оставь нужную версию (или напиши новую), удали маркеры конфликта. Сохрани файл.

**7. Заверши слияние:**
```
git add greeting.js
git commit
```
(Сообщение коммита будет предложено автоматически, можно оставить как есть.)

**8. Проверь историю:**
```
git log --oneline --graph
```
Ты увидишь merge commit с двумя родителями.

### Откат: «эксперимент был тупиком»

**kind:** `text**

Представь: ты провёл эксперименты, создал конфликт, разрешил его... и понял, что всё это было зря. Экспериментальные ветки и файлы не нужны. Нужно вернуться к состоянию `main` до всех экспериментов — как будто ничего не было.

Сейчас ты этому научишься.

**1. Посмотри историю `main`:**
```
git log --oneline
```
Найди глазами **последний коммит до экспериментов**. Это тот, который был в `main` до fast-forward merge с `test-merge.js`. Обычно это предпоследний или третий с конца. Запомни его хэш (первые 7 символов).

**2. Откати `main` к этому коммиту:**
```
git reset --hard <хэш>
```
Замени `<хэш>` на тот, что ты нашёл.

`git reset --hard` перемещает указатель ветки назад на указанный коммит. Коммиты выше этой точки становятся недостижимыми (пропадают из `git log`), но физически ещё хранятся в Git — их можно восстановить через `git reflog`. Файлы в рабочей папке возвращаются к состоянию на момент этого коммита.

**3. Проверь, что эксперименты исчезли:**
```
ls
```
Файлов `test-merge.js` и `greeting.js` больше нет.

```
git log --oneline
```
Коммитов эксперимента тоже нет — история `main` чистая.

**4. Удали экспериментальные ветки:**
```
git branch -D feat/greeting-update
git branch -D experiment/merge-lab
```

**5. Убедись, что остались только нужные ветки:**
```
git branch
```
Должны быть `main` и `feat/substring-search`.

**Важное предупреждение:** `git reset --hard` переписывает историю. Это безопасно только пока ты не запушил изменения на GitHub. Если ветка уже опубликована — используй `git revert` (создаёт коммит-отмену, не трогая историю). Но поскольку мы экспериментировали локально, `reset` — правильный инструмент.
