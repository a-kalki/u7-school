# Слияние веток и конфликты

**Краткое содержание:**
Урок посвящён операции `git merge` — слиянию веток. Студент изучает fast-forward и merge-commit, учится разрешать конфликты слияния, а затем откатывает все эксперименты — как будто пошли не туда и решили вернуться.

### 1. Что такое merge и зачем он нужен

**Merge (слияние)** — это объединение изменений из одной ветки в другую. Когда ты закончил работу в feature-ветке, ты сливаешь её в `main`, чтобы изменения стали частью основного кода.

### 2. Fast-forward merge

**Fast-forward** — самый простой случай слияния. Он происходит, когда в целевой ветке (`main`) не было новых коммитов с момента создания feature-ветки.

```
main:     A---B
               \
feat/X:         C---D
```

После `git merge feat/X` (находясь в `main`):

```
main:     A---B---C---D
feat/X:         C---D
```

Git просто «перематывает» указатель `main` вперёд на последний коммит ветки `feat/X`. Коммиты `C` и `D` становятся частью истории `main`.

### 3. Merge commit (трёхстороннее слияние)

Если в `main` появились новые коммиты после создания feature-ветки, Git создаёт **merge commit** — специальный коммит, у которого два родителя:

```
main:     A---B---E
               \
feat/X:         C---D
```

После `git merge feat/X`:

```
main:     A---B---E---M
               \     /
feat/X:         C---D
```

`M` — merge commit, который объединяет изменения из `E` и `D`. Если изменения не конфликтуют, Git делает это автоматически.

### 4. Конфликты слияния

**Конфликт** возникает, когда одна и та же строка (или блок кода) изменена в обеих ветках по-разному. Git не может решить, какую версию оставить — он просит помощи у разработчика.

Как выглядит конфликт в файле:

```
<<<<<<< HEAD
function greet() {
  console.log('Привет из main!');
=======
function greet() {
  console.log('Привет из feat/X!');
>>>>>>> feat/X
}
```

**Обозначения:**
- `<<<<<<< HEAD` — начало конфликтного блока (твоя текущая ветка)
- `=======` — разделитель между версиями
- `>>>>>>> feat/X` — конец конфликтного блока (сливаемая ветка)

**Как разрешить конфликт:**
1. Открой конфликтные файлы
2. Выбери нужную версию (или напиши новую, объединяющую обе)
3. Удали маркеры `<<<<<<<`, `=======`, `>>>>>>>`
4. Сохрани файл
5. `git add .` — отметить конфликт как разрешённый
6. `git commit` — завершить слияние

### 5. Откат: когда эксперимент зашёл в тупик

Иногда эксперимент не удался, и проще откатить всё назад, чем разбираться в конфликтах. Два основных способа:

**Способ 1 — отмена незавершённого слияния:**

Если конфликт ещё не разрешён и не закоммичен:
```bash
git merge --abort
```
Возвращает репозиторий к состоянию до `git merge` — как будто ничего не было.

**Способ 2 — откат ветки к предыдущему состоянию:**

Если изменения уже закоммичены и нужно «отмотать» ветку назад:
```bash
git reset --hard HEAD~N   # откат на N коммитов назад
git reset --hard <хэш>    # откат к конкретному коммиту
```

`git reset --hard` **переписывает историю ветки**: указатель ветки перемещается на заданный коммит. Коммиты выше этой точки становятся недостижимыми из ветки (их не видно в `git log`), но физически остаются в объектной базе Git — их можно восстановить через `git reflog` в течение ~30 дней. Позже сборщик мусора (`git gc`) удалит их окончательно.

Это безопасно только для локальных веток, которые ты ещё не запушил. Для веток, доступных другим (например, `main` на GitHub), лучше использовать `git revert` — он создаёт новый коммит-отмену, не трогая историю.

**Разница между reset и revert:**

| | `git reset --hard` | `git revert` |
|---|---|---|
| История | Переписывается | Сохраняется |
| Новые коммиты | Нет | Да (коммит-отмена) |
| Когда использовать | Локально, до push | После push, в shared-ветках |
| Опасность | Можно потерять работу | Безопасно |

### 6. Практика: основные команды

Слияние ветки `feat/X` в текущую:
```bash
git merge feat/X
```

Если возник конфликт, Git скажет: `CONFLICT (content): Merge conflict in файл.js`

Посмотреть, какие файлы конфликтуют:
```bash
git status
```

Отменить слияние (если передумал):
```bash
git merge --abort
```
