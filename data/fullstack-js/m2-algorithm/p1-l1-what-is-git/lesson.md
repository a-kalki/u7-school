# Что такое Git и зачем он нужен

**Краткое содержание:**
Урок знакомит с проблемой версионирования кода и её решением — системой контроля версий Git. Студент узнает ключевые понятия (репозиторий, коммит, stage, ветка), установит Git и выполнит базовую настройку.

### Подробный конспект

#### 1. Проблема: жизнь без системы контроля версий

Представь, что ты пишешь программу. Ты сохраняешь файл, вносишь изменения, снова сохраняешь. Через неделю понимаешь — то, что ты сделал три дня назад, было лучше. Но старой версии уже нет — ты её перезаписал.

**Типичные «решения» без VCS:**
- `index.js`, `index_v2.js`, `index_final.js`, `index_final_final.js`
- `project/`, `project_backup/`, `project_backup_2/`
- Файлы, залитые в Telegram/Google Drive с датами в названии

Всё это приводит к путанице: где актуальная версия? Кто и когда что менял? Как откатить изменения?

#### 2. Что такое система контроля версий (VCS)

**VCS (Version Control System)** — инструмент, который отслеживает каждое изменение в твоём проекте. Git — самая популярная VCS.

Git запоминает **состояние файлов** в каждый момент времени. Ты сам решаешь, какие изменения сохранить как «контрольную точку» — **коммит (commit)**.

#### 3. Ключевые понятия Git

- **Репозиторий (repository)** — папка проекта, за которой следит Git. Все данные Git хранятся в скрытой папке `.git` внутри проекта.
- **Коммит (commit)** — сохранённое состояние файлов в конкретный момент. У каждого коммита есть уникальный идентификатор (хеш), автор и сообщение.
- **Stage (индекс)** — «зона подготовки» перед коммитом. Ты выбираешь, какие изменения попали в следующий коммит, а какие — нет.
- **Ветка (branch)** — независимая линия разработки. Основная ветка обычно называется `main`. Можно создавать ветки для экспериментов или новых фич, не ломая основной код.

**Аналогия с компьютерной игрой:**
- Коммит = сохранение (save) в игре
- Ветка = разные слоты сохранений (можно пробовать рискованные ходы, не портя основное прохождение)
- Stage = выбор, что именно сохранить: инвентарь, прогресс, позицию

#### 4. Установка Git

**Linux (Ubuntu/Debian):**
```bash
sudo apt install git
```

**macOS:**
```bash
brew install git
```
Или установить Xcode Command Line Tools: `xcode-select --install`.

**Windows:**
Скачать с [git-scm.com](https://git-scm.com/download/win).

**Проверка установки:**
```bash
git --version
# Ожидаемый вывод: git version 2.x.x
```

#### 5. Базовая настройка Git

Перед первым использованием Git нужно представиться — указать имя и email. Они будут записываться в каждый коммит:

```bash
git config --global user.name "Твоё Имя Фамилия"
git config --global user.email "твой.email@example.com"
```

Флаг `--global` означает, что настройка применится для всех репозиториев на этом компьютере (глобально). Если нужно указать разные данные для конкретного проекта — используй те же команды без `--global` внутри папки проекта.

**Проверка настроек:**
```bash
git config --global user.name
git config --global user.email
```

#### 6. Почему Git — must-have для разработчика

1. **История изменений** — всегда видно, кто, когда и зачем что-то поменял
2. **Откат** — можно вернуться к любому предыдущему состоянию
3. **Параллельная работа** — ветки позволяют нескольким людям работать над разными фичами одновременно
4. **Резервная копия** — при использовании GitHub/GitLab твой код хранится не только локально
5. **Код-ревью** — коллеги могут смотреть твои изменения до того, как они попадут в основной код
