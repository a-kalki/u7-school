# Приглашение ревьюеров и правила ревью

**Краткое содержание:**
Студент учится добавлять ревьюеров в PR, изучает правила конструктивного код-ревью и требования модуля (2 аппрува).

### 1. Зачем нужно код-ревью

Ты написал код, тесты проходят, всё работает. Зачем его ещё кому-то показывать?

**Код-ревью решает несколько проблем:**

- **Свежий взгляд** — ты смотришь на код несколько часов и перестаёшь замечать ошибки. Ревьюер видит код впервые и замечает то, что ты пропустил
- **Обучение** — ревьюер предлагает более удачные решения, а ты учишься. И наоборот — ревьюя чужой код, ты видишь разные подходы
- **Общее знание** — если код смотрели два человека, то в случае отпуска/болезни одного второй понимает, как что работает
- **Стандарты** — ревьюеры следят, чтобы код соответствовал правилам команды (именование, структура, тесты)

### 2. Как добавить ревьюеров в PR

1. Открой свой PR на GitHub
2. В правой панели найди раздел **Reviewers**
3. Нажми на шестерёнку или строку поиска
4. Введи GitHub-username ревьюера
5. Выбери из выпадающего списка

**Правило модуля:** нужно одобрение **двух** ревьюеров (2 аппрува), чтобы PR можно было слить (merge). Это не просто формальность — каждый ревьюер смотрит на код со своей стороны и может заметить разные проблемы.

### 3. Правила код-ревью в нашей команде

#### Тон комментариев

**Предложение** (не критическое):
- Формулируется как предложение: «**Предлагаю** сделать так, потому что...», «**Можно** переименовать переменную, потому что...»
- Это не требование — автор PR может согласиться или аргументированно отказаться
- Пример: «Предлагаю вынести проверку типов в отдельную функцию, потому что она повторяется в нескольких местах»

**Критическое замечание:**
- Формулируется как требование: «**Нужно** исправить, потому что...», «**Необходимо** добавить проверку, потому что...»
- Тон — настойчивый, но уважительный. Объясняем причину
- Пример: «Нужно добавить проверку на undefined в len(), потому что если передать undefined, цикл уйдёт в бесконечность»

**Как отличить предложение от критического замечания:**
- **Предложение** — код работает правильно, но можно сделать лучше (читаемость, стиль, производительность)
- **Критическое замечание** — код содержит ошибку, уязвимость или нарушает правила, без исправления которых нельзя принимать PR

#### Итог ревью

- **Approve** (одобрить) — ревьюер не нашёл критических замечаний. Все комментарии — предложения, автор сам решает, что учесть
- **Request changes** (запросить правки) — есть хотя бы одно критическое замечание. Это **блокирующее** ревью: PR нельзя слить, пока автор не исправит замечания и ревьюер не подтвердит исправления

**Простыми словами:**
- Если ревьюер написал только «Предлагаю...» и «Можно...» — ставит **Approve** ✅
- Если ревьюер написал хотя бы одно «Нужно исправить...» — ставит **Request changes** ❌

#### Что смотреть при ревью

Ревьюер проверяет код по нескольким направлениям:

1. **Корректность** — правильно ли работает код? Все ли тестовые случаи покрыты?
2. **Граничные случаи** — что будет с пустой строкой? С отрицательным числом? С `undefined`?
3. **Соответствие правилам** — не используются ли запрещённые методы (`.length`, `.slice()` и т.д.)?
4. **Читаемость** — понятны ли имена переменных и функций? Нет ли избыточного кода?
5. **Тесты** — покрывают ли тесты основные и граничные случаи?

### 4. Как принимать ревью

Когда ты получаешь комментарии ревьюера — это не критика личности, а профессиональная обратная связь. Твоя задача — спокойно прочитать, понять и либо исправить, либо аргументированно ответить.

**Пример ответа на предложение:**
> Ревьюер: «Предлагаю переименовать `temp` в `currentChar`, потому что название не отражает смысл переменной»
> Ты: «Согласен, переименовал» — и пушишь правку

**Пример ответа на критическое замечание:**
> Ревьюер: «Нужно добавить проверку на пустую строку в isEqual, иначе функция падает с TypeError»
> Ты: «Исправил, добавил проверку в начале функции» — и пушишь правку

**Пример аргументированного отказа:**
> Ревьюер: «Предлагаю использовать цикл for-of вместо обычного for»
> Ты: «Не могу, потому что for-of использует итератор массива, а по правилам модуля нам нужен ручной обход по индексам»
