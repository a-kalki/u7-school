# Ветки в Git

**Краткое содержание:**
Урок знакомит с концепцией веток (branches) в Git. Студент узнаёт, зачем нужны ветки, как их создавать, переключаться, удалять и что происходит с коммитами при удалении.

### 1. Зачем нужны ветки

Представь, что ты работаешь над новой фичей, но тут приходит срочный баг-фикс. Если ты продолжишь в той же ветке, ты смешаешь стабильный код и незаконченную фичу. Результат: бардак и неработающий код.

**Ветка** — это «параллельная вселенная» для кода. Ты можешь:
- Создать ветку под новую задачу и работать в ней независимо
- Ветка `main` (или `master`) остаётся стабильной и готовой к релизу
- Когда работа в ветке завершена — ты сливаешь (merge) изменения обратно в `main`

**Аналогия:**
- `main` — чистовик тетради с уже проверенными задачами
- Feature-ветка — черновик, где ты решаешь новую задачу
- Если решение получилось — переписываешь в чистовик. Если нет — выбрасываешь черновик, чистовик не пострадал

### 2. Основные команды

Просмотр списка веток:
```bash
git branch
```
Текущая ветка отмечена звёздочкой `*`.

Создание новой ветки:
```bash
git branch feat/новая-фича
```

Переключение на ветку:
```bash
git switch feat/новая-фича
```
Или (старый способ):
```bash
git checkout feat/новая-фича
```

Создать и сразу переключиться (шорткат):
```bash
git switch -c feat/новая-фича
```

### 3. Удаление веток

Когда ветка больше не нужна, её можно удалить:

```bash
git branch -d feat/новая-фича
```

Флаг `-d` (delete) удалит ветку, только если она уже слита в другую ветку — это защита от случайной потери работы. Если Git говорит, что ветка не слита — используй `-D` (принудительно):

```bash
git branch -D feat/новая-фича
```

**Нельзя удалить ветку, в которой ты находишься.** Перед удалением переключись на другую:

```bash
git switch main
git branch -d feat/новая-фича
```

**Что происходит с коммитами при удалении ветки?**

Ветка — это всего лишь указатель (ярлык) на коммит. Удаление ветки убирает ярлык, но не удаляет сами коммиты.

- Если коммиты **слиты** в другую ветку (например, `main`) — они остаются доступны через историю `main`. Всё в порядке.
- Если коммиты **не слиты** — они становятся «висячими» (dangling). Git не показывает их в `git log`, но они какое-то время хранятся во внутренней базе. Git периодически чистит «висячие» коммиты сборщиком мусора (`git gc`), и тогда они удаляются безвозвратно.

**Аналогия:** ветка — закладка в книге. Убрать закладку ≠ вырвать страницы. Страницы (коммиты) остаются на месте. Но если на эти страницы больше никто не ссылается — со временем их могут «утилизировать».

### 4. Именование веток

Принято использовать осмысленные имена. Распространённые префиксы:
- `feat/` — новая функциональность (например, `feat/substring-search`)
- `fix/` — исправление бага (например, `fix/login-error`)
- `chore/` — хозяйственные задачи (например, `chore/update-deps`)

**Правила хорошего тона:**
- Никаких `my-branch`, `test1`, `temp` — через неделю не поймёшь, что это
- Используй дефисы или слеши для разделения слов: `feat/user-authentication`
- В больших проектах добавляют ID задачи: `feat/JIRA-123-add-login`

### 5. Как ветки хранятся в Git

Технически, ветка — это просто **указатель на коммит**. Когда ты создаёшь ветку, Git не копирует файлы — он просто запоминает: «ветка `feat/X` сейчас на коммите `abc123`». Когда ты делаешь новый коммит в этой ветке, указатель сдвигается на новый коммит.

Именно поэтому создание и переключение веток происходит мгновенно даже в больших проектах.

### 6. HEAD — указатель на текущую ветку

`HEAD` — специальный указатель, который говорит Git, на какой ветке ты сейчас находишься. Когда ты переключаешь ветку через `git switch`, `HEAD` перемещается на новую ветку.

Проверить, куда смотрит HEAD:
```bash
git branch
```
Ветка с `*` — та, на которую указывает HEAD.
