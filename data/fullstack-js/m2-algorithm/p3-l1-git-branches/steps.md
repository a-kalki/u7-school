### Шаг 2 — Эксперимент в отдельной ветке

**kind:** `text`

> **Windows:** все команды ниже выполняй в **Git Bash** (устанавливается вместе с Git). Обычный cmd или PowerShell не подойдут — в них команды `echo`, `ls` и другие работают иначе.

Сейчас ты проведёшь эксперимент в **отдельной** ветке, чтобы не затронуть рабочие файлы. Перейди в папку `js-algorithms`.

**1. Посмотри текущие ветки:**
```
git branch
```
Видна только `main` (или `master`).

**2. Создай экспериментальную ветку и сразу переключись в неё:**
```
git switch -c experiment/test-branch
```

**3. Убедись, что ты в экспериментальной ветке:**
```
git branch
```
Звёздочка `*` должна быть на `experiment/test-branch`.

**4. Создай файл и закоммить его в экспериментальной ветке:**
```
echo "Эксперимент с ветками" > experiment.txt
git add experiment.txt
git commit -m "Эксперимент: добавляю файл в experiment/test-branch"
```

**5. Проверь, что файл существует:**
```
ls
```

### Шаг 3 — Переключение: почувствуй «параллельность»

**kind:** `text`

**1. Переключись обратно на `main`:**
```
git switch main
```

**2. Посмотри содержимое папки:**
```
ls
```
Файла `experiment.txt` **нет**! Git хранит его только в коммитах ветки `experiment/test-branch`. Ветка `main` об этом коммите ничего не знает — она указывает на другую точку истории.

**3. Посмотри историю в `main`:**
```
git log --oneline
```
Коммита «Эксперимент...» здесь нет.

**4. Переключись обратно в экспериментальную ветку:**
```
git switch experiment/test-branch
```

**5. Проверь:**
```
ls
git log --oneline
```
Файл вернулся вместе с веткой. Коммит виден.

**Вывод:** каждая ветка — независимое состояние проекта. Переключение между ними меняет рабочую папку под историю коммитов этой ветки.

### Шаг 4 — Удаление экспериментальной ветки

**kind:** `text`

Эксперимент окончен, ветка больше не нужна.

**1. Переключись на `main` (нельзя удалить ветку, в которой находишься):**
```
git switch main
```

**2. Попробуй удалить экспериментальную ветку:**
```
git branch -d experiment/test-branch
```
Git откажется: ветка не слита в `main`, он боится что ты потеряешь работу. Это защита.

**3. Удали принудительно:**
```
git branch -D experiment/test-branch
```

**4. Проверь, что ветка исчезла:**
```
git branch
```
Осталась только `main`.

**5. Что стало с коммитом «Эксперимент...»?**

Выполни:
```
git log --oneline
```
Коммита нет — он не был никуда слит и стал «висячим» (dangling). Git его запомнил во внутренней базе, но не показывает. Со временем сборщик мусора (`git gc`) удалит его безвозвратно.

Попробуй вернуть его к жизни (опционально, для понимания):
```
git reflog
```
Ты увидишь запись о коммите. `reflog` — это локальный журнал всего, что ты делал. По нему можно восстановить «потерянные» коммиты в течение ~30 дней.

**Вывод:** ветка — это всего лишь ярлык на коммит. Удаление ветки ≠ удаление коммитов. Но если коммиты никуда не слиты и ярлыков на них не осталось — они становятся мусором.

### Шаг 5 — Создание рабочей ветки для проекта

**kind:** `text`

Теперь, когда ты понимаешь механику, создай рабочую ветку для реальной задачи — реализации функций поиска подстрок.

```
git switch -c feat/substring-search
```

Проверь:
```
git branch
```
Звёздочка на `feat/substring-search`. Эта ветка будет жить весь проект 3 — именно в ней ты реализуешь функции и будешь делать push для Pull Request.
