# Модули: import и export

## Зачем нужны модули

Представь, что ты пишешь программу из 1000 строк в одном файле `app.js`. Найти нужную функцию — боль. Две переменные `result` в разных местах конфликтуют. Добавить новую фичу страшно — можно сломать старое.

**Модули решают эти проблемы:**

- **Разделение**: код разбит на файлы по смыслу. `math.js` — вычисления, `db.js` — работа с данными, `app.js` — логика приложения.
- **Изоляция**: переменные и функции модуля не видны снаружи, пока ты явно их не экспортируешь.
- **Переиспользование**: модуль `math.js` можно импортировать в десятке других файлов.

```javascript
// 📁 math.js — живёт своей жизнью
const SECRET = 'не видно снаружи';

export function sum(a, b) {
  return a + b;
}

// 📁 app.js
import { sum } from './math.js';
console.log(sum(2, 3)); // 5
// console.log(SECRET); — ошибка! SECRET не экспортирован
```

В Bun модули — это стандарт. Любой `.js`-файл — это модуль. Каждый файл изолирован, импорты работают «из коробки», без настроек.

## Named export — экспорт нескольких сущностей

Именованный экспорт (`named export`) позволяет отдать наружу несколько значений из одного модуля. Можно экспортировать переменные, функции, классы.

```javascript
// 📁 math.js

// Экспорт прямо при объявлении
export const PI = 3.14159;

export function sum(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

// Или отдельно — после объявления
const E = 2.71828;
export { E };
```

Один модуль может иметь **сколько угодно** именованных экспортов. Имена важны — импортёр будет обращаться по ним.

---

## Named import — импорт по имени

Импортёр указывает фигурные скобки `{}` и перечисляет нужные имена:

```javascript
// 📁 app.js
import { sum, multiply, PI } from './math.js';

console.log(sum(2, 3));      // 5
console.log(multiply(2, 3)); // 6
console.log(PI);             // 3.14159
```

Можно импортировать **всё сразу** как объект:

```javascript
import * as math from './math.js';

console.log(math.PI);          // 3.14159
console.log(math.sum(2, 3));   // 5
```

`import * as` удобен когда экспортов много и ты хочешь видеть «привязку»: `math.sum()` сразу говорит, откуда функция.

**Важно:** в зависимости от настроек проекта использовать расширение файла `.js` в импортах может быть обязательно — `'./math.js'`, не `'./math'`. Но обычно не требуется.

## Default export и import

У модуля может быть **один** экспорт по умолчанию (`default`). При импорте имя можно выбрать любое — фигурные скобки не нужны.

```javascript
// 📁 user.js
export default function createUser(name) {
  return { name: name };
}

// 📁 app.js
import makeUser from './user.js';   // имя может отличаться от оригинального!
const aliya = makeUser('Алия');
console.log(aliya.name);            // 'Алия'
```

Обрати внимание: в `user.js` функция называется `createUser`, а при импорте мы назвали её `makeUser`. С default-экспортом имя не имеет значения — импортёр сам решает как назвать.

**Смешанный импорт** — default + named в одной строке:

```javascript
// 📁 user.js
export default function createUser(name) {
  return { name: name };
}
export function formatName(user) {
  return user.name.toUpperCase();
}

// 📁 app.js
import makeUser, { formatName } from './user.js';
const user = makeUser('Алия');
console.log(formatName(user));     // 'АЛИЯ'
```

| Named export | Default export |
|---|---|
| Несколько на модуль | Один на модуль |
| `import { имя }` — фигурные скобки | `import имя` — без скобок |
| Имя фиксировано | Имя выбирает импортёр |
| Удобно для библиотек утилит | Удобно для главной сущности модуля |

Многие команды предпочитают не использовать `default`-экспорт. Причина: с named-экспортом имя всегда одинаковое во всех файлах — легче искать, проще рефакторить, автодополнение в редакторе работает точнее.

## Реэкспорт и barrel-файлы

Когда модулей становится много, удобно собрать их в одном файле-агрегаторе — **barrel-файле** (часто `index.js`). Он ничего не делает сам, только реэкспортирует.

```javascript
// 📁 lib/index.js  ← barrel-файл
export { sum, multiply } from './math.js';
export { default as createUser } from './user.js';
```

Теперь импортёр подключает один файл вместо двух:

```javascript
// 📁 app.js
import { sum, createUser } from './lib/index.js';
```

**Плюсы:** чистая точка входа, меньше путей в импортах.
**Минус:** если переусердствовать — грузится всё сразу, даже ненужное.

## Порядок загрузки модулей

Bun разрешает зависимости рекурсивно перед выполнением кода:

```
app.js импортирует math.js и user.js
  math.js — нет зависимостей
  user.js импортирует math.js (уже загружен — используется кеш)
```

Порядок выполнения:
1. `math.js` — загружается первым (от него зависят все)
2. `user.js` — после math
3. `app.js` — после всех зависимостей

Код **верхнего уровня** модуля (вне функций) выполняется сразу при импорте. Код внутри функций — только при вызове.

```javascript
// 📁 math.js
console.log('[math] верхний уровень');   // выполняется при импорте

export function sum(a, b) {
  console.log('[math] sum вызвана');     // выполняется при вызове
  return a + b;
}
```

### Кеширование модуля

Когда Bun впервые встречает `import`, он выполняет код модуля и **запоминает** результат — это и есть кеширование. При повторном импорте того же модуля (из любого другого файла) код **не выполняется заново** — возвращается уже готовый объект экспорта.

```javascript
// 📁 counter.js
console.log('[counter] загружен');   // этот лог появится только ОДИН раз
export let count = 0;
export function inc() { count++; }

// 📁 a.js
import { count, inc } from './counter.js';
inc();
console.log('a: count =', count);   // 1

// 📁 b.js
import { count } from './counter.js';
console.log('b: count =', count);   // 1 — тот же count!
```

Вывод при запуске `a.js` и `b.js` в одном процессе:
```
[counter] загружен    ← один раз
a: count = 1
b: count = 1
```

**Что это значит на практике:**
- Все импортёры **разделяют одно и то же состояние** экспортированных переменных
- `console.log` на верхнем уровне модуля сработает только при первом импорте
- Если модуль меняет свою переменную — изменение видно всем, кто его импортирует
- Это удобно для общего состояния (например, конфигурации), но может быть источником трудноуловимых ошибок
