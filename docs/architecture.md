# Архитектура платформы u7-school

## Обзор

Платформа **u7-school** построена по модульному принципу. Приложение разделёно на доменные модули,
каждый из которых инкапсулирует свою предметную область. Взаимодействие с модулями происходит
через команды (commands), что позволяет единообразно подключать разные интерфейсы: консоль,
Telegram-бота и REST API.

## Доменные модули

Приложение делится на независимые доменные модули (core, w3school).

Каждый модуль — это отдельный npm-пакет в директории `packages/`.

## Двухслойная архитектура модуля

Каждый модуль внутри разделён на три слоя:

```
packages/<module>/
  src/
    domain/       # Слой предметной области
    api/          # Слой приложения
    ui/           # Слой интерфейса пользователя 
```

### Слой domain

Содержит **типы**, **схемы валидации**, **агрегаты** и **политики**. Это чистый код предметной
области без привязки к инфраструктуре.

- **Типы (Entity):** Структуры данных предметной области, выведенные из Valibot-схем.
- **Схемы (Schema):** Valibot-схемы для валидации инвариантов.
- **Агрегаты (Ar):** Классы, инкапсулирующие состояние сущности и поведение (богатая доменная
  модель). Принимают состояние сущности через конструктор, предоставляют методы для его изменения
  с проверкой инвариантов.
  - Инварианты проверяются standalone-функцией `validateInvariants(entity)`, которая вызывается
    в конструкторе (при создании из существующего состояния) и в фабричном методе `create()`
    (при создании нового агрегата). Это исключает дублирование и гарантирует проверку при
    любом способе создания.
- **Политики (Policy):** Объекты, проверяющие права доступа (создание, чтение, редактирование,
  удаление). Импортируются напрямую там, где требуется проверка прав.

### Слой api

Содержит **репозитории** и **сценарии использования (Use Cases)**. Оркеструет доменные объекты
для выполнения бизнес-операций.

- **Use Case (Uc):** Каждый сценарий использования — отдельный класс/функция, реализующая одну
  бизнес-операцию (например, создание пользователя).
- **Repository:** Интерфейс для сохранения и извлечения агрегатов. Реализации подменяются
  (in-memory для тестов, БД для продакшена).

Структура папок:
```
```
- use-cases: объекты use-case;
- repositories: репозитории;
- controllers: точки входа в api:
    - console: логика консольного взаимодействия;
    - bot: логика взаимодействия через бота;
    - ui: логика взаймодествия через rest;
```
- module.ts
```

### Вход в модуль: команды и entry point

Каждый модуль предоставляет единый **entry point** — класс, которая принимает
команду и возвращает результат:

```typescript
// packages/core/src/api/module.ts
module.handle(command: ModuleCommand): Promise<Result>;
```

### Обработка исключений

Иерархия исключений:
- **`AppException`** (абстрактный базовый класс):
  - Поля: `name`, `userMessage`, `debugInfo`, `httpStatus`, `payload`
  - Метод `rest()` → `{ error, message, status, payload }` для HTTP-ответа
  - **`DomainException`** (level: `"domain"`) — ошибки предметной области:
    - Нарушение инвариантов (validation)
    - Нарушение бизнес-правил, включая права доступа на основе ролей (accessDenied)
  - **`ApiException`** (level: `"api"`) — ошибки уровня приложения:
    - Аутентификация, невалидный токен, сессия (accessDenied)

Фабричные методы для типовых ошибок:
- `DomainException.validation(userMsg, debug, payload?)` — 422, payload для ошибок валидации
- `DomainException.conflict(userMsg, debug)` — 409, конфликт (дубликат)
- `DomainException.notFound(entity, id)` — 404, сущность не найдена
- `DomainException.accessDenied(userMsg, debug)` — 403, нарушение доменных прав (роль, владелец)
- `ApiException.accessDenied(userMsg, debug)` — 403, ошибки уровня приложения (не авторизован, невалидный токен)

Уровень контроллера самостоятельно перехватывает ошибки и решает что далее делать с ними.


## Соглашения об именовании

| Концепт | Шаблон имени | Пример |
|---|---|---|
| Сущность (тип) | `<Имя>` | `User`, `Course` |
| Схема валидации | `<Имя>Schema` | `UserSchema`, `CourseSchema` |
| Агрегат | `<Имя>Ar` | `UserAr`, `CourseAr` |
| Политика прав | `<Имя>Policy` | `UserPolicy`, `CoursePolicy` |
| Объект БД (опционально) | `<Имя>Record` | `UserRecord`, `CourseRecord` |
| Объект вывода (опционально) | `<Имя>Output` / `<Имя>Out` | `UserOutput`, `UserOut` |
| Команда + схема | `<Действие><Имя>Command` + `<Действие><Имя>CommandSchema` | `CreateUserCommand`, `CreateUserCommandSchema` |
| Сценарий использования | `<Имя><Действие>Uc` | `UserCreatingUc`, `CourseCreatingUc` |

### Структура команды

Каждая команда размещается в `api/commands/` и состоит из:
- **Типа** (TypeScript-тип, выведенный из Valibot-схемы)
- **Схемы валидации** (Valibot-схема для строгой проверки входящих данных)

Схема команды переиспользует поля схемы сущности (`UserSchema.entries.name`) —
это позволяет держать единственный источник истины для правил валидации.

**Правило:** все DTO (команды, запросы), зависящие от доменных сущностей, **обязаны**
наследовать правила валидации из схем сущностей через `Schema.entries`. Не дублировать
пайпы валидации в DTO.

Пример:
```typescript
// packages/core/src/api/commands/create_user_command.ts
import * as v from "valibot";
import { UserSchema } from "../../domain/user/user";

export const CreateUserCommandSchema = v.object({
  name: UserSchema.entries.name,
  telegramId: UserSchema.entries.telegramId,
  role: UserSchema.entries.role,
});

export type CreateUserCommand = v.InferOutput<typeof CreateUserCommandSchema>;
```

Обёрточная команда модуля — plain object со следующей структурой:

```typescript
type ModuleCommand = {
  name: string;          // Имя команды, например "create-user"
  user?: string;         // UUID пользователя, инициировавшего команду (опционально)
  attrs: unknown;        // Параметры команды
};
```

### Валидационный хелпер `parseOrThrow`

Универсальный хелпер `api/shared/parse_or_throw.ts`:
- Принимает Valibot-схему и значение
- При ошибке выбрасывает `DomainException.validation` с плоскими issues
- При успехе возвращает типизированный output
- Используется в командах, инвариантах агрегатов и других местах, где нужна строгая валидация

### Поток выполнения команды

```
```
Console / Bot / REST: ModuleCommand { name, user?, attrs } -->
Entry point модуля `handle(command)`: Диспатчер команд модуля -->
Use Case: Проверка валидации команды (Valibot-схемa), Проверка прав (Policy), Изменение через агрегат (Ar), Сохранение (Repo) -->
Result
```
```

## Межмодульное взаимодействие

- **Изменение, чтение состояния:** Только через фасад модуля или сам модуль (entry point + команду). Прямой вызов Use Case другого модуля запрещён.
- **Доменные объекты:** Доменные объекты (сущности, схемы, агрегаты, политики) можно импортировать напрямую из любого модуля с соблюдением модульной иерархии зависимостей. Команды, use-case и репозитории импортируются только через entry point модуля.

## Тестирование

### Изоляция тестов
- Тесты **не должны зависеть** друг от друга. Каждый тест создаёт своё состояние.
- Использовать `beforeEach` или создавать свежие экземпляры в каждом тесте.
- Поощряется создание хелпер-функций (например `addUser(repo, overrides)`) для сокращения кода.
- Целевой размер теста: **≤10 строк**. Допустимо до 20, в исключительных случаях до 30.

### Проверка ошибок
- Тесты исключений **обязаны** проверять конкретное сообщение: `.toThrow("ожидаемый текст")`.
  Проверка только класса (`.toThrow(DomainException)`) запрещена — один класс покрывает
  validation, conflict, notFound, accessDenied, и тест должен различать их.

### Тестирование агрегатов vs схем
- **Схемы валидации:** детальные тесты на все поля и граничные случаи.
- **Агрегаты:** один тест, подтверждающий что инварианты подключены (проверка через схему).
  Остальные тесты агрегата фокусируются на бизнес-логике (методы изменения, граничные случаи).

### Моки репозиториев
- В тестах use-case допускается мокать репозитории, используя in-memory реализации.

## Инверсия зависимостей (DI)

Модуль принимает при создании объект `ModuleResolve`, и предоставляет его объектам use-case. Этот объект должен быть явно определен типом, чтобы клиенты точно знали что можно получить из нее.

```typescript
// Пример интерфейса DI-контейнера
type CoreResolve = {
  userRepo: UserRepository;
  courseRepo: CourseRepository;
}
```
