# Архитектура платформы u7-school

## Обзор

Платформа **u7-school** построена по модульному принципу. Бэкенд разделён на доменные модули,
каждый из которых инкапсулирует свою предметную область. Взаимодействие с модулями происходит
через команды (commands), что позволяет единообразно подключать разные интерфейсы: консоль,
Telegram-бота и REST API.

## Доменные модули

Приложение делится на независимые доменные модули:

- **core** — базовые сущности платформы: пользователи, курсы, уроки, модули.
- **w3school** — интеграция с w3school: парсинг, обогащение, конвертация контента.

Каждый модуль — это отдельный npm-пакет в директории `packages/`.

## Двухслойная архитектура модуля

Каждый модуль внутри разделён на два слоя:

```
packages/<module>/
  src/
    domain/       # Слой предметной области
    api/          # Слой приложения (use cases)
    index.ts      # Entry point модуля
```

### Слой domain

Содержит **типы**, **схемы валидации**, **агрегаты** и **политики**. Это чистый код предметной
области без привязки к инфраструктуре.

- **Типы (Entity):** Структуры данных предметной области, выведенные из Valibot-схем.
- **Схемы (Schema):** Valibot-схемы для валидации инвариантов.
- **Агрегаты (Ar):** Классы, инкапсулирующие состояние сущности и поведение (богатая доменная
  модель). Принимают состояние сущности через конструктор, предоставляют методы для его изменения
  с проверкой инвариантов.
- **Политики (Policy):** Объекты, проверяющие права доступа (создание, чтение, редактирование,
  удаление). Импортируются напрямую там, где требуется проверка прав.

### Слой api

Содержит **репозитории** и **сценарии использования (Use Cases)**. Оркеструет доменные объекты
для выполнения бизнес-операций.

- **Use Case (Uc):** Каждый сценарий использования — отдельный класс/функция, реализующая одну
  бизнес-операцию (например, создание пользователя).
- **Repository:** Интерфейс для сохранения и извлечения агрегатов. Реализации подменяются
  (in-memory для тестов, БД для продакшена).

## Соглашения об именовании

| Концепт | Шаблон имени | Пример |
|---|---|---|
| Сущность (тип) | `<Имя>` | `User`, `Course` |
| Схема валидации | `<Имя>Schema` | `UserSchema`, `CourseSchema` |
| Агрегат | `<Имя>Ar` | `UserAr`, `CourseAr` |
| Политика прав | `<Имя>Policy` | `UserPolicy`, `CoursePolicy` |
| Объект БД (опционально) | `<Имя>Record` | `UserRecord`, `CourseRecord` |
| Объект вывода (опционально) | `<Имя>Output` / `<Имя>Out` | `UserOutput`, `UserOut` |
| Команда | `<Действие><Имя>Command` | `CreateUserCommand`, `CreateCourseCommand` |
| Сценарий использования | `<Имя><Действие>Uc` | `UserCreatingUc`, `CourseCreatingUc` |

## Вход в модуль: команды и entry point

Каждый модуль предоставляет единый **entry point** — экспортируемую функцию, которая принимает
команду и возвращает результат:

```typescript
// packages/core/src/index.ts
export function handle(command: ModuleCommand): Promise<Result>;
```

### Структура команды

Команда — plain object со следующей структурой:

```typescript
type ModuleCommand = {
  name: string;          // Имя команды, например "create-user"
  user?: string;         // UUID пользователя, инициировавшего команду (опционально)
  attrs: unknown;        // Параметры команды
};
```

### Поток выполнения команды

```
Console / Bot / REST
        │
        ▼
  ModuleCommand { name, user?, attrs }
        │
        ▼
  ┌─────────────────────────────┐
  │  Entry point модуля         │
  │  handle(command)            │
  └──────────┬──────────────────┘
             │
             ▼
  ┌─────────────────────────────┐
  │  Диспатчер команд           │
  │  (map name → Use Case)      │
  └──────────┬──────────────────┘
             │
             ▼
  ┌─────────────────────────────┐
  │  Use Case (Uc)              │
  │  ┌───────────────────────┐  │
  │  │ Проверка прав (Policy)│  │
  │  │ Создание агрегата (Ar)│  │
  │  │ Сохранение (Repo)     │  │
  │  └───────────────────────┘  │
  └──────────┬──────────────────┘
             │
             ▼
         Result
```

## Межмодульное взаимодействие

- **Изменение состояния:** Только через фасад модуля (entry point + команду). Прямой вызов
  Use Case другого модуля запрещён.
- **Чтение данных:** Через фасад модуля.
- **Проверка прав:** Policy-объекты можно импортировать напрямую из любого модуля. Это
  допустимо, так как политики — stateless-функции без побочных эффектов.

## Инверсия зависимостей (DI)

Elysia не предоставляет встроенного DI-контейнера. В проекте используется простой подход:

- **Контейнер зависимостей** — `Map<string, unknown>` в корне модуля.
- При инициализации модуля в контейнер регистрируются реализации репозиториев, политик и других
  зависимостей.
- Use Cases получают зависимости из контейнера.
- В тестах контейнер перезаписывается in-memory реализациями.

```typescript
// Пример интерфейса DI-контейнера
interface Container {
  register<T>(key: string, instance: T): void;
  resolve<T>(key: string): T;
}
```

## Файловая структура модуля (на примере core)

```
packages/core/
  src/
    domain/
      user/
        user.ts          # User (тип), UserSchema (схема)
        roles.ts         # Role (enum), RoleSchema
        user_ar.ts       # UserAr (агрегат)
        user_policy.ts   # UserPolicy (политика)
      course/
        course.ts        # Course, CourseSchema
        course_ar.ts     # CourseAr
        course_policy.ts # CoursePolicy
      ...
    api/
      user/
        user_creating_uc.ts         # UserCreatingUc
        user_repository.ts          # UserRepository (интерфейс)
        in_memory_user_repository.ts # InMemoryUserRepository
      course/
        course_creating_uc.ts
        course_repository.ts
        in_memory_course_repository.ts
      di/
        container.ts     # DI-контейнер
    index.ts             # Entry point + диспатчер
```
